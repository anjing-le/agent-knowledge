# 知识库项目 - 面试 Q&A 准备清单

## 一、项目层面

### Q1: 介绍一下你的项目？

**A:** 我做的是一个基于 RAG（检索增强生成）技术的智能知识库系统。用户可以上传 PDF、Word 等文档到知识库，系统会自动解析文档、分块、向量化，然后在问答时通过向量检索找到最相关的文档片段，结合 LLM 生成精准回答。整个项目前端用 Vue 3 + Element Plus，后端用 Spring Boot 3 + JPA，文档解析用 Python FastAPI 独立服务，属于一个完整的全栈 AI 应用。

---

### Q2: 为什么选择做知识库项目？

**A:** 主要有两个原因。一是大模型虽然很强，但它不知道企业内部的专有知识，比如产品手册、技术文档这些内容。RAG 可以让大模型"读懂"这些文档再回答，解决了知识时效性和私有化的问题。二是 RAG 是目前大模型应用落地最主流的技术方案，涉及文档解析、向量化、检索、生成多个环节，做一遍下来对整个 AI 应用开发链路的理解会很深。

---

### Q3: 项目的核心功能有哪些？

**A:** 核心功能有四个：一是知识库管理，支持创建、编辑、删除知识库，还可以配置 Embedding 模型、分块大小、重叠大小这些 RAG 参数；二是文档管理，支持上传 PDF/Word/Excel/TXT 文档，系统自动走解析→分块→向量化的处理流水线；三是智能问答，用户提问后系统先做向量检索找到相关文档片段，再送给 LLM 生成回答，并且会标注内容来源；四是对话管理，支持多轮对话，保留最近 10 轮历史上下文。

---

### Q4: RAG 是什么？为什么要用 RAG？

**A:** RAG 全称 Retrieval-Augmented Generation，检索增强生成。核心思想是在调用大模型之前，先从知识库中检索出和用户问题相关的文档片段，把这些片段作为上下文一起送给 LLM。这样做的好处是：第一，大模型能回答它训练数据中没有的内容；第二，回答有据可查，减少幻觉；第三，不需要微调模型，只需要更新知识库文档就能更新知识。比如在我的项目中，用户问"履约管理是什么"，系统会先检索到相关文档片段，再让 LLM 基于这些内容回答。

---

### Q5: 文档解析是怎么做的？支持哪些格式？

**A:** 文档解析用的是独立的 Python FastAPI 服务（doc-parser），支持 PDF、Word、Excel、TXT 四种格式。后端通过 `DocParserClient` 发 HTTP 请求把文件发过去，doc-parser 解析完返回纯文本内容。之所以用独立服务而不是在 Java 里解析，是因为 Python 在文档解析方面的库生态更成熟，比如 PDF 解析有 pypdfium2，Word 解析有 python-docx。整个调用是同步的，解析完直接进入下一步分块。

---

### Q6: 文档分块（Chunk）是怎么实现的？为什么要分块？

**A:** 分块是因为大模型有 Token 长度限制，一篇几万字的文档不可能整个塞进去，而且检索时也需要更细粒度的匹配。我用的是固定窗口加重叠的策略，默认每块 500 字、重叠 50 字。重叠是为了避免语义被截断——比如一句话刚好在分界点，加了重叠后前后两块都能包含完整语义。具体代码在 `DocumentProcessingService.simpleChunking()` 里，会按字符数切分，同时做句子边界优化，尽量不在句子中间断开。分块大小和重叠大小都是可配置的，存在知识库的 `chunkSize` 和 `chunkOverlap` 字段里。

---

### Q7: 向量化（Embedding）是什么意思？怎么实现的？

**A:** 向量化就是把文本转成一组数字（向量），这样就可以用数学方法计算两段文字的语义相似度。我用的是 `text-embedding-3-small` 模型，通过 OpenAI 兼容接口调用，生成 1536 维的向量。具体在 `EmbeddingService` 里，支持单条和批量两种方式，批量处理时每 20 条一批发送请求。每个文档分块向量化后，向量和分块 ID 一起存入内存向量库 `VectorStoreService`。向量化用的模型是可配置的，存在知识库实体的 `embeddingModel` 字段里。

---

### Q8: 向量检索是怎么做的？用的什么算法？

**A:** 用的是余弦相似度算法。检索时先把用户的问题也做向量化，然后和向量库里的所有向量逐一计算余弦相似度。余弦相似度衡量的是两个向量方向上的一致性，值域是 -1 到 1，越接近 1 越相似。具体实现在 `VectorStoreService.cosineSimilarity()` 里，就是两个向量点积除以各自模的乘积。检索时先取 Top-20 候选，再用相似度阈值 0.3 过滤掉不相关的，最后返回 Top-5。整个向量库用 `ConcurrentHashMap` 实现，按知识库 ID 分组存储。

---

### Q9: 检索结果怎么和 LLM 结合生成回答的？

**A:** 在 `LLMService.buildRAGSystemPrompt()` 里，我会把检索到的文档片段格式化成"参考内容"，和系统提示词一起发给 LLM。提示词里有几个关键要求：第一，严禁幻觉，必须忠于原文；第二，宁缺毋滥，没有相关信息就直接说"没找到"而不是编造；第三，做相关性判断，如果检索到的内容和问题其实不相关，也要如实告知；第四，回答必须标注来源文档名称。另外还会带上最近 10 轮对话历史，以支持多轮对话。最终消息顺序是 system prompt → 历史消息 → 用户新问题。

---

### Q10: 项目的整体架构是怎样的？

**A:** 三层架构。前端是 Vue 3 + Element Plus + TypeScript，负责知识库管理、文档上传和智能问答界面。后端是 Spring Boot 3 + JPA + MySQL，是核心业务层，包括知识库 CRUD、文档处理流水线（解析→分块→向量化）、向量检索、LLM 调用和对话管理。文档解析服务是独立的 Python FastAPI（doc-parser），专门负责各种格式文档的文本提取。三者通过 HTTP API 通信。数据存储用 MySQL 存元数据，内存 ConcurrentHashMap 存向量。

---

### Q11: 前后端是怎么分工的？

**A:** 前端负责页面展示和用户交互，后端负责全部业务逻辑。前端通过 Axios 调后端 REST API，开发环境用 Vite 的 proxy 解决跨域。后端暴露三组 API：知识库管理（`/api/knowledge`）、文档管理（`/api/knowledge/{kbId}/documents`）、智能问答（`/api/chat`）。前端有几个核心页面：知识库列表、知识库详情（文档管理 + RAG 配置展示）、文档切片查看、智能问答。后端核心服务包括 DocumentProcessingService（处理流水线）、RetrievalService（检索）、ChatService（对话）、LLMService（大模型调用）。

---

### Q12: 数据库设计了哪些表？为什么这样设计？

**A:** 核心表有五张。`knowledge_base` 存知识库信息和 RAG 配置（分块大小、Embedding 模型等）；`document` 存文档元数据和处理状态；`chunk` 存分块内容、向量 ID 和向量化状态；`conversation` 存会话信息和关联的知识库 ID 列表；`message` 存每条对话消息、角色和引用来源。这样设计的思路是把 RAG 的每个环节都持久化——文档的处理状态可以追踪，分块和向量的关系可以管理，对话历史可以回溯。另外还有 `file_storage` 表做文件存储管理，支持文件去重（MD5 校验）。

---

### Q13: 如果文档很大，分块策略怎么优化？

**A:** 当前用的是固定窗口策略，比较简单。如果要优化，有几个方向：第一，语义分块，按段落、章节等自然边界切分，而不是机械按字数切；第二，递归分块，先按大块切分，如果某块太长再递归细分；第三，滑动窗口加更大重叠，增加上下文连续性；第四，对于结构化文档（比如表格、代码），用专门的分块策略保持结构完整。在我的项目中，分块大小和重叠大小已经做成了可配置的，存在知识库实体里，前端也能看到和修改。

---

### Q14: 检索准确率不高怎么办？有什么优化思路？

**A:** 几个方向。第一，引入 Rerank（重排序），先粗检索 Top-20 候选，再用交叉编码器对 query 和每个候选做精排，准确率会明显提升。第二，混合检索，向量检索和关键词检索（BM25）结合，互补短处。第三，Query 改写，用 LLM 把用户的口语化问题改写成更精确的检索 query。第四，调优分块策略和相似度阈值。在我的项目中，RetrievalService 已经实现了相似度阈值过滤（默认 0.3），过低的匹配会被过滤掉，并且日志会打印分数范围，方便调优。

---

### Q15: 如果要支持多用户，架构需要怎么调整？

**A:** 当前项目没有做多租户隔离。要支持多用户的话，主要改三个地方：第一，所有表加 `user_id` 或 `tenant_id` 字段做数据隔离，查询时带上用户条件；第二，向量库也要按用户隔离，当前 `VectorStoreService` 是按知识库 ID 分组的 ConcurrentHashMap，可以直接复用这个结构，只要保证不同用户的知识库 ID 不冲突即可；第三，API 层加认证和鉴权，用 JWT 或 Session 方案。另外如果用户量大，内存向量库要换成 Milvus 或 Qdrant 等专业向量数据库。

---

### Q16: 内存向量库重启后数据不就丢了吗？怎么解决的？

**A:** 对，因为用的是 ConcurrentHashMap 存向量，进程重启确实会丢。我的解决方案是实现了一个 `VectorIndexRebuilder` 组件，用 Spring 的 `@EventListener(ApplicationReadyEvent.class)` 监听应用启动事件，启动完成后自动从数据库里查出所有已启用知识库的已向量化分块，批量重新调用 Embedding API 生成向量，再写入内存向量库。这样每次重启后，向量数据会自动恢复。这是教学项目的权衡——生产环境应该用 Milvus 等持久化向量数据库。

---

### Q17: 文档处理为什么要用异步？遇到过什么问题？

**A:** 因为文档解析、分块、向量化整个流程可能要几十秒甚至几分钟，如果同步处理，用户上传文档后就会一直等着，体验很差。所以我用了 Spring 的 `@Async` 注解让处理逻辑在后台线程执行，上传接口立即返回。但遇到过一个坑：`@Async` 方法在事务提交前就开始执行了，导致查不到刚插入的文档记录。解决方案是用 `TransactionSynchronizationManager.registerSynchronization()` 的 `afterCommit` 回调，确保事务提交后再触发异步处理。

---

### Q18: 项目有什么不足？如果继续做会怎么改进？

**A:** 主要有几个不足。第一，向量存储用的内存，不适合大规模数据，应该换成 Milvus 或 Qdrant；第二，分块策略比较简单，只有固定窗口，应该支持语义分块、递归分块等多种策略；第三，没有 Rerank 重排序，检索精度还有提升空间；第四，LLM 调用是同步的，没有做流式输出（SSE），用户体验可以更好。如果继续做，我会优先加 Rerank 和流式输出，然后考虑接入专业向量数据库。

---

## 二、大模型应用层面

### Q1: 大模型（LLM）是什么？有什么特点？

**A:** 大模型是基于 Transformer 架构、在海量文本上预训练的语言模型，比如 GPT-4、GPT-4o-mini。它的特点是：理解和生成自然语言能力强，能做翻译、摘要、问答、代码生成等多种任务；通过 Prompt 就能控制行为，不需要针对每个任务单独训练；但也有局限，比如知识有截止日期、容易产生幻觉、不知道企业私有信息。在我的项目中用的是 gpt-4o-mini，通过 OpenAI 兼容接口调用。

---

### Q2: Prompt 是什么？怎么写好一个 Prompt？

**A:** Prompt 就是发给大模型的指令文本，模型根据 Prompt 来生成回答。写好 Prompt 的关键是：明确角色（你是什么）、明确任务（要做什么）、给出约束（不能做什么）、提供格式要求。比如在我的项目里，RAG 的系统提示词明确要求"严禁幻觉，必须忠于原文"、"宁缺毋滥，没有信息就说没找到"、"回答前加 ✅/⚠️/❌ 标识"、"必须标注来源文档名"。好的 Prompt 能显著提升回答质量。

---

### Q3: Token 是什么？为什么要关注 Token 数量？

**A:** Token 是大模型处理文本的最小单位，大约 1 个中文字对应 1-2 个 Token，1 个英文单词对应 1-3 个 Token。需要关注 Token 数量因为：第一，模型有上下文窗口限制，比如 4K、8K、128K Token，超了就处理不了；第二，API 调用按 Token 计费，输入和输出都要算钱；第三，输入 Token 越多，延迟越大。所以 RAG 里做分块的一个目的就是控制送入 LLM 的文本量，在我的项目中默认取 Top-5 相关片段，避免 Token 超限。

---

### Q4: Temperature 参数是什么意思？

**A:** Temperature 控制模型输出的随机性。值在 0-1 之间，越低输出越确定、越保守，越高越随机、越有创意。Temperature=0 时模型几乎总是选概率最高的下一个 Token，适合事实性问答；Temperature=1 时输出更多样，适合创作。在我的项目中设的是 0.7，因为 RAG 场景需要一定的组织能力来整合检索内容，但又不能太随机以免编造信息。这个参数也是可配置的，存在 `application.yml` 的 `app.llm.temperature` 里。

---

### Q5: RAG 和微调（Fine-tuning）有什么区别？各自适用场景？

**A:** RAG 是在推理时把外部知识检索出来作为上下文给模型参考，不改变模型本身；微调是用特定数据重新训练模型参数，让模型"学会"新知识。RAG 的优势是：不需要 GPU 训练、知识可以实时更新（改文档就行）、成本低；劣势是检索质量影响最终效果。微调的优势是模型直接掌握知识，回答更自然；劣势是需要高质量训练数据、GPU 成本高、知识更新慢。一般来说，企业知识库、FAQ 这种知识频繁更新的场景用 RAG 更合适，风格化写作、特定领域术语等场景更适合微调。我的项目选 RAG 就是因为知识库文档随时可能更新。

---

### Q6: Embedding 模型和生成模型有什么区别？

**A:** Embedding 模型的任务是把文本编码成一个固定维度的向量（比如 1536 维），用于衡量语义相似度，输出是一组数字；生成模型的任务是根据输入生成新文本，输出是自然语言。两者的训练目标不同：Embedding 模型训练的是"相似文本的向量应该接近"，生成模型训练的是"预测下一个 Token"。在我的项目中，Embedding 用的是 `text-embedding-3-small`，生成用的是 `gpt-4o-mini`，两个模型各司其职，Embedding 负责检索环节，生成模型负责回答环节。

---

### Q7: 什么是幻觉（Hallucination）？RAG 怎么减少幻觉？

**A:** 幻觉是指大模型生成了看起来合理但实际上不正确或无中生有的内容。比如你问一个它不知道的事实，它可能编一个听起来很像的答案。RAG 减少幻觉的核心思路是"给模型证据"——把检索到的真实文档片段作为参考，要求模型基于这些内容回答。在我的项目中还做了几层防护：Prompt 里明确要求"绝对禁止幻觉"、"忠于原文"；检索内容和问题不相关时让模型说"没找到"而不是编答案；回答标注来源文档名，方便用户核实；前端展示引用原文和相似度分数。

---

### Q8: 什么是 Rerank？为什么需要 Rerank？

**A:** Rerank 是在向量检索之后做的二次精排。向量检索用的是双塔模型，query 和文档分别编码再比较，速度快但精度有限；Rerank 用的是交叉编码器，把 query 和每个候选文档拼在一起送入模型做精确打分，精度更高但速度慢。所以通常先用向量检索快速召回 Top-20 候选，再用 Rerank 精排取 Top-5。在我的项目中目前没有实现 Rerank，但 RetrievalService 的架构已经预留了位置，检索流程是 `向量搜索 → [Rerank] → 过滤和限制`，后续可以很方便地加上。

---

### Q9: 余弦相似度为什么适合用来衡量文本相似度？

**A:** 因为余弦相似度衡量的是两个向量的方向一致性，而不是绝对距离。文本 Embedding 的特点是——语义相似的文本在高维空间中方向接近，但由于文本长度不同，向量的模（长度）可能差异很大。余弦相似度只看方向不看长度，所以不受文本长短影响。比如"猫"和"一只可爱的小猫"语义相近，虽然 Embedding 向量的模可能不同，但方向一致，余弦相似度就会很高。在我的项目中，余弦相似度的计算就是标准的点积除以模的乘积。

---

### Q10: 大模型应用和传统应用开发有什么区别？

**A:** 最大的区别是"不确定性"。传统应用是确定性的——相同输入必定相同输出，可以写单元测试精确验证。大模型应用的输出是概率性的，每次可能不一样，而且可能出错（幻觉）。这带来几个挑战：第一，测试方式不同，不能简单断言输出，要评估输出质量；第二，需要额外的工程手段来保证可靠性，比如 RAG 用检索兜底、Prompt 加约束、前端标注来源让用户核实；第三，成本模型不同，按 Token 计费，要考虑控制输入输出长度；第四，性能特点不同，LLM 调用通常要几秒，需要做异步或流式处理。

---

### Q11: 你觉得大模型应用的难点在哪里？

**A:** 我觉得主要在三个方面。第一是效果可控性，模型可能幻觉、跑题、格式不对，需要通过精心设计 Prompt、多层过滤、人工兜底来保证质量。第二是工程复杂性，RAG 涉及文档解析、分块、向量化、检索、生成等多个环节，每个环节都有很多参数要调优，比如分块大小、相似度阈值、Top-K、Temperature 等，组合爆炸。第三是评估困难，怎么量化"回答好不好"没有标准答案，不像传统应用有明确的对错判断。在我的项目中，我用日志打印相似度分数范围、来源标注等方式来辅助评估。

### Q12: 如果让你设计一个 AI 产品，你会怎么考虑？

**A:** 我会按这个思路来。第一，明确场景——AI 要解决什么具体问题，不是所有场景都需要大模型。第二，选对技术路线——如果是知识问答用 RAG，如果是风格化输出考虑微调，如果是复杂任务流程考虑 Agent。第三，做好 Prompt 工程——这是性价比最高的优化手段，好的 Prompt 能省掉很多后续工程。第四，考虑可靠性——加兜底策略、错误处理、来源追溯。第五，控制成本——选择合适大小的模型、控制 Token 用量、做缓存。这些都是我在做知识库项目过程中体会到的。

### Q13: 向量数据库和传统数据库有什么区别？

**A:** 传统数据库（如 MySQL）存储的是结构化数据，查询方式是精确匹配或条件筛选，比如 `WHERE name = 'xxx'`。向量数据库存储的是高维向量，查询方式是相似度计算，找"最像的"而不是"完全一样的"。向量数据库的核心能力是近似最近邻搜索（ANN），用 HNSW、IVF 等索引算法在亿级向量中快速找到最相似的 Top-K。在我的项目中，为了教学简化用了内存 ConcurrentHashMap 做暴力搜索，生产环境应该用 Milvus 或 Qdrant 这样的专业向量数据库来支撑大规模数据和持久化。
