## 1. 开场锚点（30秒，一句话抓住注意力）

### 风格A：好奇驱动型 ⭐推荐

> "我在用 ChatGPT 的时候发现了一个有意思的问题——它什么都能聊，但你问它你们公司的退货政策是什么，它就开始瞎编了。我就想，能不能让大模型先'翻书'再回答？于是我做了一个基于 RAG 技术的智能知识库系统，让 AI 能读懂你上传的文档，再基于文档内容精准回答问题。"

**为什么这样开场**：老师喜欢有好奇心的学生。从"发现问题→产生疑问→动手解决"这个链条切入，自然而不做作。

### 风格B：问题驱动型

> "大模型有个天生的短板——它的知识有截止日期，也不知道企业内部的私有数据。怎么让大模型回答它本来不知道的东西？业界的主流方案叫 RAG，检索增强生成。我就动手做了一个完整的 RAG 知识库系统，从文档解析到向量检索到智能问答，把整条链路都跑通了。"

**为什么这样开场**：直接点出行业痛点，展示你对技术趋势的了解。适合面对偏工程导向的导师。

### 风格C：对比驱动型

> "一个比喻——普通 ChatGPT 像闭卷考试，全靠记忆，记错了就编答案。我做的这个系统相当于让 AI 变成开卷考试，先从你的文档里找答案，再组织语言回答，还会标注'这段话出自哪个文件'。这就是 RAG 技术的核心思路。"

**为什么这样开场**：用生活化比喻让任何背景的老师都能秒懂，而且"闭卷 vs 开卷"的对比很直观，容易记住。

### 深潜1：RAG（检索增强生成）

🎯 **通俗比喻**

RAG 就是让 AI 参加"开卷考试"。普通 ChatGPT 是闭卷——全靠记忆，记不住就编。RAG 是开卷——先翻书找到相关内容，再根据书上的内容回答问题，每个答案都有出处。

📖 **技术本质**

RAG 全称 Retrieval-Augmented Generation，分两步：第一步"检索"（Retrieval），把用户的问题和知识库里的文档做语义匹配，找出最相关的片段；第二步"生成"（Generation），把这些片段作为上下文塞进 Prompt，让大模型基于这些真实内容生成回答。核心价值是让大模型能回答它训练数据里没有的内容，同时减少幻觉。

💭 **我的理解**

> "我觉得 RAG 最精妙的地方在于它的'分工'——检索模块负责找对资料，生成模块负责组织语言。就像一个好的研究助手，先帮你把论文找出来摊在桌上，然后你再根据论文写综述。这样做的好处是知识更新很轻，我不需要重新训练模型，只要往知识库里上传新文档就行了。在做这个项目的过程中，我最大的体会是 RAG 看起来思路简单，但每个环节的质量都会影响最终效果——分块太粗检索不准，分块太细又丢失上下文，这些 trade-off 让我对 AI 工程有了更深的理解。"

❓ **可能追问**

- **"RAG 和直接把文档全塞给 GPT 有什么区别？"**
  → 上下文窗口有限制（比如 128K Token），长文档塞不下；而且全塞进去检索效率很低、成本也高。RAG 先筛选出最相关的几个片段，精准且省钱。
- **"RAG 有什么局限性？"**
  → 检索质量是瓶颈。如果用户问的方式和文档写法差异很大，向量检索可能召回不到正确的片段。可以通过 Rerank、Query 改写、混合检索来优化。

### 深潜2：Embedding / 向量化

🎯 **通俗比喻**

Embedding 就像是给每段文字发一个"GPS坐标"。不同的是这个坐标在一个1536维的空间里。意思相近的话，坐标就靠在一起——"退货条件"和"退款政策"的坐标很近，但和"今天天气"的坐标就隔得很远。

📖 **技术本质**

Embedding 是一种将离散的文本映射到连续的高维向量空间的技术。通过预训练的 Embedding 模型（如 text-embedding-3-small），将一段文字编码为1536维的浮点数向量。模型的训练目标是让语义相似的文本对应的向量在空间中距离更近（方向更一致）。这个向量本身没有人类可读的含义，但它编码了文本的语义信息。

💭 **我的理解**

> "向量化让我最震撼的一点是——机器终于不是在做字符串匹配了，而是在理解'意思'。你搜'怎么退货'，它能找到写着'退款流程'的文档，哪怕两个句子没有一个字重复。在实现的时候，我注意到一个细节：问题向量化和文档向量化必须用同一个 Embedding 模型，否则生成的向量不在同一个空间里，算出来的相似度就没有意义了。这个项目里我把 Embedding 模型做成了知识库级别的配置项，保证同一个知识库下的文档和查询用的是同一个模型。"

❓ **可能追问**

- **"1536维是什么概念？为什么需要这么多维？"**
  → 维度越高，模型能编码的语义信息越丰富。就像用2个数字只能表示平面上的点，用3个能表示空间中的点。1536维是模型设计者在"信息容量"和"计算成本"之间选的平衡点。
- **"Embedding 模型是怎么训练出来的？"**
  → 核心思路是对比学习。把语义相似的文本对（比如问答对）作为正样本，让模型学习把它们编码得更近；随机的文本对作为负样本，编码得更远。训练目标是让模型学会"什么是相似"。

---

### 深潜3：余弦相似度

🎯 **通俗比喻**

想象两支箭射出去——余弦相似度看的不是两支箭飞了多远，而是它们指向的方向是不是一致的。方向完全一致就是1，完全相反就是-1，毫无关系（垂直）就是0。在 RAG 里，我们用它来判断用户的问题和文档片段"聊的是不是同一件事"。

📖 **技术本质**

余弦相似度 = 两个向量的点积 ÷ 两个向量的模之积，即 cos(θ) = (A·B) / (|A|×|B|)。它衡量的是向量方向的一致性，范围 [-1, 1]。在文本检索中特别合适，因为不同长度的文本 Embedding 后向量的模可能差异很大，但余弦相似度只看方向，不受模长影响。

💭 **我的理解**

> "选余弦相似度而不是欧氏距离，是我在做的时候特意查过的。欧氏距离衡量的是'两点之间有多远'，余弦相似度衡量的是'两个方向有多像'。文本场景下，一篇长文章的向量模可能比一个短句子大很多，用欧氏距离会偏向短文本。但在语义上，长文章和短句子完全可以是'说的同一件事'，所以用余弦相似度更合理。在我的项目中，相似度的计算代码其实不长，就是公式直接实现，但它背后的数学直觉是我花了时间去理解的。"

❓ **可能追问**

- **"余弦相似度 0.3 作为阈值是怎么定的？"**
  → 经验值。太低（比如0.1）会引入很多不相关内容，太高（比如0.8）又可能错过有关但措辞不同的片段。0.3 在实际测试中过滤噪声效果不错。我还在日志里打印了过滤前后的分数范围，方便调优。
- **"除了余弦相似度还有什么度量方式？"**
  → 欧氏距离（L2距离）和点积（内积）。OpenAI 的 Embedding 模型生成的向量是归一化的（模长为1），这时候余弦相似度和点积的结果是一样的。

### 深潜4：文档分块策略

🎯 **通俗比喻**

一本500页的书，你不会整本丢给别人让他找答案。你会先把书拆成一章一章、一段一段，然后告诉他"答案大概在第三章第二段"。分块就是把长文档拆成 AI 能处理的小段落的过程。

📖 **技术本质**

分块（Chunking）是将长文本切分为固定大小的片段，每个片段独立进行向量化和检索。关键参数有两个：chunk_size（每块多大，默认500字）和 overlap（相邻块的重叠区域，默认50字）。重叠是为了防止语义在块的边界处被截断——比如一句完整的话刚好在分界点，有了重叠，前后两个块都能包含这句话的完整语义。

💭 **我的理解**

> "分块大小是我在这个项目中反复调过的一个参数。太大的块检索时定位不够精准——你想找'退货需要几天'，结果返回了一整页的退货政策；太小的块又会丢失上下文——一个块里可能只有半句话，向量化后语义不完整。500字加50字重叠是我试出来的一个平衡点。另外我在分块的时候还做了一个小优化——尽量在句号或换行符处截断，而不是机械地按字数切，这样每个块至少是一个完整的句子。我还把分块参数做成了知识库级别的配置，前端可以可视化编辑，修改后对文档点'重新解析'就能用新参数重新分块。"

❓ **可能追问**

- **"除了固定窗口分块，还有什么策略？"**
  → 语义分块（按段落/标题等自然边界切分）、递归分块（先粗切再细分）、还有针对结构化内容（表格/代码）的专用分块策略。固定窗口最简单但最通用，其他策略更精细但实现复杂。
- **"重叠太大会怎样？太小呢？"**
  → 重叠太大（比如和块大小差不多）会导致冗余高、向量库膨胀、检索时同一段内容反复出现。太小（比如0）就会出现语义断裂。一般取块大小的10%左右作为重叠。

### 深潜5：向量数据库 vs 传统数据库

🎯 **通俗比喻**

传统数据库像图书馆的索引卡——你按书名、作者、ISBN精确查找，要么找到要么没有。向量数据库像一个"语义雷达"——你说"我想看关于太空探险的小说"，它不是按字面找"太空探险"这四个字，而是找到所有和太空探险"意思接近"的书，按接近程度排个序推荐给你。

📖 **技术本质**

传统关系型数据库（MySQL）存储结构化数据，用精确匹配（WHERE = / LIKE）做查询。向量数据库（Milvus、Qdrant、Weaviate）存储高维向量，用近似最近邻搜索（ANN）做查询，核心是在海量向量中快速找到与查询向量最相似的 Top-K 个。生产级向量数据库用 HNSW、IVF 等索引结构来加速搜索，避免暴力遍历。

💭 **我的理解**

> "在我的项目里，我没有用 Milvus 这样的专业向量数据库，而是用了 Java 的 ConcurrentHashMap 在内存里存向量，检索时暴力遍历计算余弦相似度。这是一个刻意的设计取舍——教学场景下数据量小（几百个文档片段），暴力搜索完全够用，而且学生能看到完整的相似度计算过程，不会被向量数据库的黑盒封装挡住。但我清楚知道这在生产环境下不可行——百万级向量暴力搜索太慢了，需要 ANN 索引。而且内存数据库重启就丢了，所以我实现了一个启动时自动重建的机制。"

❓ **可能追问**

- **"ANN（近似最近邻）为什么叫'近似'？"**
  → 因为追求精确最近邻需要遍历所有向量，O(n)太慢了。ANN 通过构建索引（如 HNSW 图结构）来加速搜索，速度可以快几百倍，但可能会漏掉少量真正相似的结果，是一个精度和速度的 trade-off。
- **"你的内存方案能支撑多大数据量？"**
  → 粗估一下：一个向量1536维 × 4字节 ≈ 6KB，加上文本内容大约10KB。1万条向量大约100MB。百万级就要几十GB了，内存装不下。所以数据量大了一定要用专业向量数据库。

### 深潜6：幻觉问题（Hallucination）

🎯 **通俗比喻**

幻觉就是 AI 在"一本正经地胡说八道"。你问它一个它不知道的事情，它不会说"我不知道"，而是编一个听起来特别合理的答案。就像一个学生考试不会也不留白，硬写一个看起来像答案的东西。

📖 **技术本质**

幻觉产生的根源是语言模型的工作方式——它本质上是在做"下一个词的概率预测"，它会生成统计上最可能出现的文本，而不是"最正确的"文本。当模型没有可靠信息时，它仍然会生成流畅的文本，因为流畅性是它被训练出来的核心能力。RAG 通过提供真实的参考内容来约束模型，让它"有据可依"而不是"自由发挥"。

💭 **我的理解**

> "幻觉是我做这个项目最关注的问题。因为知识库问答如果出现幻觉，用户看到的是一个看起来很权威、标注了来源的错误答案，危害比普通聊天严重得多。我在 Prompt 里做了三层防护：第一，明确告诉模型'绝对禁止幻觉，必须忠于原文'——这是最高优先级的指令；第二，要求模型做相关性判断——即使检索到了内容，如果和问题其实不相关，也要回答'❌ 未找到相关信息'，而不是硬凑答案；第三，要求标注真实的来源文档名，让用户可以回溯核实。效果确实比不加约束好很多，但也不是百分百能杜绝。我觉得这个问题本质上是当前大模型架构的局限。"

❓ **可能追问**

- **"除了 Prompt 约束，还有什么方法减少幻觉？"**
  → 后处理校验（检查回答中的信息是否能在原文中找到）、Temperature 调低让输出更确定、多次生成取一致结果、让模型自我反思检查。还有一种思路是训练时加入"拒绝回答"的能力。
- **"你觉得幻觉能完全消除吗？"**
  → 我个人觉得在当前 Transformer 架构下很难完全消除，因为模型的本质是概率生成。但可以做到大幅降低，让幻觉频率低到可接受的程度，再配合人类审核来兜底。

---

### 深潜7：Rerank（重排序）

🎯 **通俗比喻**

向量检索像用渔网捞鱼——一网下去捞上来20条，里面有大鱼有小鱼还有几根水草。Rerank 就是一个经验丰富的渔夫，逐条看过来，把真正值钱的鱼挑出来排好序。第一步求快（能不能捞上来），第二步求准（是不是你要的）。

📖 **技术本质**

Rerank 使用交叉编码器（Cross-Encoder）对 query 和每个候选文档做精细打分。与 Embedding 检索不同——Embedding 是"双塔模型"，query 和文档分别独立编码再比较；交叉编码器是把 query 和文档拼在一起输入模型，模型能看到两者的交互信息，因此打分更准确，但速度更慢。典型流程：向量检索召回 Top-20 → Rerank 精排取 Top-5。

💭 **我的理解**

> "Rerank 我在项目里没有实现，但我了解它的原理和价值。我觉得它的精妙之处在于'分层'思想——先用快但粗的方法圈定候选范围，再用慢但准的方法精选。这在信息检索领域其实是个经典范式。在我的项目里，RetrievalService 的流程是'向量搜索 → 阈值过滤 → 取TopK'，架构上已经预留了 Rerank 的位置，如果要加的话，就是在阈值过滤之前插一个重排步骤。如果继续做下去，这是我最想加的功能。"

❓ **可能追问**

- **"为什么交叉编码器比双塔模型更准？"**
  → 双塔模型的 query 和文档是独立编码的，彼此"看不到对方"；交叉编码器把它们拼在一起输入，能捕捉两者之间的细粒度交互（比如某个词在 query 和文档中的不同含义），所以更准确。代价是每对 query-doc 都要过一次模型，无法预计算。

---

### 深潜8：Prompt Engineering（提示词工程）

🎯 **通俗比喻**

Prompt 就是你"怎么跟 AI 说话"。同样的模型，你随便问和精心设计问法，效果天差地别。就像同一个实习生，你说"帮我查下资料"和"帮我从这三份文件里找退货相关的段落，找到后用要点列出来，找不到就说没有"——第二种方式才能得到你想要的结果。

📖 **技术本质**

Prompt Engineering 是通过设计输入文本（提示词）来引导大模型产生期望输出的技术。核心要素包括：角色设定（你是什么）、任务描述（做什么）、约束条件（不能做什么）、输出格式（以什么形式回答）、Few-shot 示例（给几个例子参照）。在 RAG 场景中，Prompt 还要包含检索到的参考内容，以及防幻觉指令。

💭 **我的理解**

> "Prompt 设计是这个项目里我投入时间最多的部分之一。刚开始我写的 Prompt 很简单——'根据以下内容回答问题'，结果模型经常自由发挥，还会编造不存在的来源名称。后来我一版一版迭代，加入了'绝对禁止幻觉'的最高优先级指令、用真实文档名做来源标注的示例、不相关时的拒绝模板，效果才稳定下来。最让我印象深刻的一个细节是——如果你在 Prompt 末尾再提醒一次'不要编造'，模型的幻觉率会明显降低。所以我在 System Prompt 的结尾加了一句'再次提醒：只使用参考内容，来源写真实文档名'。Prompt 设计确实是个实验科学，要反复试。"

❓ **可能追问**

- **"System Prompt 和 User Prompt 有什么区别？"**
  → System Prompt 设定模型的角色和行为准则，全程生效；User Prompt 是每次用户的具体问题。在 RAG 场景中，System Prompt 里包含了参考内容和防幻觉规则，User Prompt 就是用户的原始问题。
- **"Prompt 越长越好吗？"**
  → 不是。太长的 Prompt 一方面消耗更多 Token（成本和延迟），另一方面模型对超长 Prompt 的注意力会分散（"lost in the middle"现象）。要精简有力，把最重要的约束放在开头和结尾。

### 深潜9：Spring Boot 异步机制

🎯 **通俗比喻**

你去奶茶店点单，同步模式是你站在柜台等奶茶做好再走；异步模式是店员给你一个取餐号，你先去逛街，做好了叫你回来取。文档处理可能要几分钟，不能让用户干等着。

📖 **技术本质**

Spring 的 `@Async` 注解可以让方法在独立线程中执行，调用方不阻塞。但有一个经典陷阱：如果在事务方法中调用 @Async 方法，异步线程可能在事务提交前就开始执行，导致读不到当前事务写入的数据。解决方案是用 `TransactionSynchronizationManager.registerSynchronization()` 注册 `afterCommit` 回调，确保事务提交后再触发异步任务。

💭 **我的理解**

> "这个坑我踩得很深。文档上传接口里，我先创建 Document 记录并保存到数据库，然后调 @Async 方法开始处理。结果异步线程里通过 documentId 查文档，查出来是 null——因为外层事务还没 commit。这个 bug 特别隐蔽，因为本地开发时数据库快，偶尔能查到（事务提交和异步启动之间的竞态条件），上了测试环境就频繁出问题。排查了两天才定位到原因，最终用 afterCommit 回调解决。这个经验让我对 Spring 事务的传播机制和线程隔离理解深了很多。"

❓ **可能追问**

- **"@Async 的线程池怎么配置的？"**
  → Spring 默认用 SimpleAsyncTaskExecutor（每次新建线程），生产环境应该配置 ThreadPoolTaskExecutor，指定核心线程数、最大线程数、队列容量。我的项目里用的默认配置，因为并发量不大。
- **"如果异步任务失败了怎么办？"**
  → 我在 DocumentProcessingService 里做了 try-catch，失败时更新文档状态为 PARSE_FAILED / CHUNK_FAILED / EMBEDDING_FAILED，前端能看到失败状态。生产环境可以加重试机制和死信队列。

### 深潜10：JPA 实体设计

🎯 **通俗比喻**

JPA 就是一个"翻译官"——你写 Java 类，它帮你翻译成数据库表；你操作 Java 对象，它帮你翻译成 SQL 语句。你不需要手写建表语句，也不需要写大段的 JDBC 代码。

📖 **技术本质**

JPA（Java Persistence API）是 Java 的 ORM 标准，Spring Data JPA 是它的实现封装。核心概念是"实体映射"——用 `@Entity`、`@Table`、`@Column` 等注解将 Java 类映射到数据库表。通过 `spring.jpa.hibernate.ddl-auto=update` 配置，JPA 可以自动根据实体类的变化更新表结构。Repository 接口继承 `JpaRepository` 后，基本的 CRUD 操作自动实现。

💭 **我的理解**

> "我的项目有五张核心表：KnowledgeBase、Document、Chunk、Conversation、Message，还有一个 FileStorage。JPA 帮我省了大量写 SQL 的时间，尤其是建表——我只需要定义好实体类，启动时 JPA 自动建表。实体之间的关系用外键字段而不是 @ManyToOne 注解来维护，避免了 JPA 懒加载和 N+1 查询的坑。比如 Document 里存了 kbId 而不是 KnowledgeBase 对象引用，查询时显式用 Repository 方法关联。"

❓ **可能追问**

- **"JPA 的 N+1 问题是什么？怎么解决？"**
  → 查一个列表时，JPA 为每条记录的关联对象单独发一条 SQL，导致 1+N 条 SQL。解决方案有：@EntityGraph 预加载、JPQL 的 JOIN FETCH、或者像我一样用外键字段避免对象关联。
- **"ddl-auto=update 在生产环境能用吗？"**
  → 不建议。update 模式只会加字段不会删字段或改类型，可能导致表结构不一致。生产环境应该用 Flyway 或 Liquibase 做数据库迁移管理。

### 深潜11：RESTful API 设计

🎯 **通俗比喻**

REST 就是用 HTTP 动词（GET/POST/PUT/DELETE）加上名词性的 URL 来描述对资源的操作。`GET /api/books` 就是"获取书的列表"，`POST /api/books` 就是"创建一本新书"，看到 URL 就知道在干什么。

📖 **技术本质**

RESTful API 的核心原则是：以资源为中心（URL 是名词不是动词）、用 HTTP 方法表示操作、无状态（每次请求自包含所有信息）、统一的响应格式。好的 REST API 具有自描述性——URL 本身就能让人理解接口的含义和层级关系。

💭 **我的理解**

> "我的 API 设计遵循了资源嵌套的风格。知识库是顶层资源 `/api/knowledge`，文档是知识库的子资源 `/api/knowledge/{kbId}/documents`，切片是文档的子资源。对话模块单独 `/api/chat`，因为对话和知识库是多对多关系，不适合嵌套。响应格式统一用 `{code, message, data}` 包装，前端 Axios 拦截器统一处理。有一个设计我觉得做得不错——发送消息时，kbIds 作为请求参数传入而不是固定绑定在会话上，这样前端可以动态切换知识库，x 掉知识库传空数组就变成纯 LLM 对话。"

❓ **可能追问**

- **"PUT 和 PATCH 有什么区别？"**
  → PUT 是全量更新（替换整个资源），PATCH 是部分更新（只改指定字段）。比如修改知识库名称，用 PATCH 更合适，因为你不需要重新传所有字段。
- **"如果接口要同时操作多个资源怎么办？"**
  → 这不太符合严格的 REST 风格。可以设计一个"动作型"API（如 `POST /api/documents/{id}/reparse`），或者用批量操作端点。

### 深潜12：前后端分离架构

🎯 **通俗比喻**

就像餐厅的前厅和后厨。前厅（前端）负责接待客人、展示菜单、传菜；后厨（后端）负责备菜、做菜、出菜。两者通过"传菜口"（API）沟通，各管各的，前厅装修不影响后厨做菜。

📖 **技术本质**

前后端分离是将用户界面和业务逻辑解耦的架构模式。前端独立开发和部署（通常是 SPA 单页应用），后端只提供 API 接口。开发时通过代理（Vite proxy）解决跨域；生产部署时前端打包成静态文件由 Nginx 托管，API 请求通过 Nginx 反向代理转发到后端服务。

💭 **我的理解**

> "前后端分离的好处在我这个项目里体现得很明显。前端用 Vite + Vue 3，热更新非常快，UI 改动秒级生效；后端用 Spring Boot，关注业务逻辑。两个可以同时开发互不影响。开发环境用 Vite 的 proxy 配置把 `/api` 前缀的请求代理到 `localhost:10001`。前端的 Axios 封装里做了统一的请求拦截（加 token）、响应拦截（处理错误码）、Loading 状态管理。"

❓ **可能追问**

- **"跨域问题怎么解决的？"**
  → 开发环境用 Vite 的 devServer proxy 解决。如果要部署到生产，用 Nginx 做反向代理，前端和 API 走同一个域名的不同路径，就不存在跨域了。
- **"前端状态管理用了什么？"**
  → 用了 Pinia（Vue 3 官方推荐的状态管理库），主要管理知识库列表、当前会话等全局状态。

### 深潜13：降级策略

🎯 **通俗比喻**

降级就像电梯坏了改走楼梯——核心功能还能用，只是体验差一点。在 AI 系统里，如果 Embedding API 挂了或者文档解析服务挂了，系统不能直接崩掉，应该有一个"差一点但能用"的备用方案。

📖 **技术本质**

降级（Degradation/Fallback）是在依赖服务不可用时，提供备选方案保证系统可用性的策略。常见模式包括：功能降级（关闭非核心功能）、数据降级（用缓存数据替代实时数据）、服务降级（本地实现替代远程调用）。在 RAG 系统中，典型的降级点包括文档解析（Python → Java 简单提取）、向量检索（向量 → 关键词匹配）、LLM 调用（GPT-4 → 简单模板回答）。

💭 **我的理解**

> "我的项目刻意**没做降级**，这是一个设计决策。比如 doc-parser 不可用时，我直接返回错误而不是用 Java 做简单的文本提取。原因是这是一个教学项目，我希望每个环节都是真实的外部调用，学生（包括我自己）能看到完整的调用链路。但我清楚知道生产环境必须做降级——doc-parser 可以降级为 Java 端 Apache Tika 提取；向量检索可以降级为关键词搜索（BM25）；LLM 挂了可以直接返回检索到的原文片段。Embedding 服务不可用时是最棘手的，因为没有向量就无法检索，这时候可以降级为全文搜索。"

❓ **可能追问**

- **"如果是你设计生产系统，降级策略怎么定？"**
  → 按影响面分级：Embedding 不可用 → 降级为关键词搜索 + 提示用户"当前为基础搜索模式"；LLM 不可用 → 直接返回检索到的原文片段 + 提示"AI 总结暂时不可用"；doc-parser 不可用 → 降级为 Java 端简单提取 + 记录待重新解析的文档。
- **"怎么判断什么时候该降级？"**
  → 两种方式：主动健康检查（我的项目里 DocParserClient 调用前先检查 /health），和被动熔断（连续 N 次调用失败后自动切到降级方案，比如 Resilience4j 的 CircuitBreaker）。

## 4. 防雷指南（求职版补充）

### 被问算法复杂度

**问："你的向量检索时间复杂度是多少？"**

> "暴力搜索是 O(n×d)，n 是向量总数，d 是维度1536。每个向量要算1536次乘法和加法。对于几百到几千的向量量级，完全可以接受。如果到了百万级别，就需要用 ANN 索引了，比如 HNSW 可以做到 O(log n) 的查询复杂度。"

### 被问生产环境怎么办

**问："这个系统如果要上生产，你觉得要改什么？"**

> "几个关键改动：第一，向量存储从内存 ConcurrentHashMap 换成 Milvus 或 Qdrant，支持持久化和大规模数据；第二，LLM 调用加流式输出（SSE），提升用户体验；第三，加 Rerank 提升检索精度；第四，加多租户数据隔离（user_id + JWT 认证）；第五，加降级和熔断——Embedding 不可用时降级为关键词搜索；第六，文档处理管道用消息队列（如 RabbitMQ）替代 @Async，保证可靠性和可重试。"

### 被问"你们团队几个人"

> "这个项目是我独立完成的。前端、后端、Python 解析服务三个部分都是我自己开发的。设计上参考了 RAGFlow 和 Dify 的开源实现，但代码是自己写的。"

⭐ **说独立完成没问题，但不要说"我一个人做了所有"时显得很轻松——要让面试官知道这有工作量。**

### 被问性能相关

**问："系统能支撑多少并发？QPS 大概多少？"**

> "坦率说我没有做过完整的压测。从架构分析，瓶颈在两个地方：一是 Embedding API 调用（外部接口，延迟不可控，大概100-500ms）；二是 LLM 调用（通常1-5秒）。向量检索本身因为是内存操作加暴力搜索，几千条向量的情况下应该在毫秒级。如果要支撑高并发，应该给 Embedding 调用加缓存（相同文本不重复调用）、LLM 改为流式输出、向量检索换成专业向量数据库。"

⭐ **没测过就说没测过，然后从架构层面分析瓶颈，比编一个数字强一百倍。**

**向量重建策略**：重点强调"方案对比"——序列化到文件 vs 启动重建 vs 持久化向量数据库，各自优缺点，为什么选启动重建。
