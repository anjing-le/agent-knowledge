## 一、RAG 完整流程图

RAG 分为两个阶段：**数据准备阶段**（离线）和**问答阶段**（在线）。

### 1.1 数据准备阶段（Indexing）

```
                    ┌──────────────────────────────────────┐
                    │          数据准备阶段（离线）           │
                    └──────────────────────────────────────┘

  ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
  │  原始文档  │────▶│  文档解析  │────▶│  文本分块  │────▶│  向量化   │
  │ PDF/Word  │     │ doc-parser│     │  Chunk   │     │ Embedding│
  │ TXT/MD    │     │ (必需)    │     │ 可配置大小 │     │ 可配置模型│
  └──────────┘     └──────────┘     └──────────┘     └─────┬────┘
                                                           │
                                                           ▼
                                                     ┌──────────┐
                                                     │  内存     │
                                                     │  向量库   │
                                                     └──────────┘
```

**流程说明**：
1. **文档解析**：调用 doc-parser 服务从 PDF/Word/TXT 等格式中提取纯文本（无降级，必须依赖 doc-parser）
2. **文本分块**：将长文本切分为 Chunk，大小和重叠由知识库配置决定（默认 500字/50字重叠）
3. **向量化**：调用 Embedding API 将每个 Chunk 转为 1536 维浮点数向量，模型可配置
4. **存储**：将向量和原文一起存入内存向量库（重启时自动从 DB 重建）

### 1.2 问答阶段（Retrieval + Generation）

```
                    ┌──────────────────────────────────────┐
                    │          问答阶段（在线）              │
                    └──────────────────────────────────────┘

  ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ 用户提问  │────▶│ 问题向量化 │────▶│ 向量检索  │────▶│ 构建提示  │
  │"退货条件？"│     │ Embedding│     │ 余弦相似度│     │ System   │
  │          │     │          │     │ 阈值过滤  │     │ Prompt   │
  └──────────┘     └──────────┘     │ 相关性判断│     └─────┬────┘
                                    └──────────┘           │
                                                           ▼
                                                     ┌──────────┐
                                                     │ LLM 生成  │
                                                     │ 带对话历史│
                                                     │ 严格基于  │
                                                     │ 检索结果  │
                                                     └──────────┘
```

**流程说明**：
1. **问题向量化**：用同一个 Embedding 模型将用户问题转为向量
2. **向量检索**：在向量库中找到与问题最相似的 Top-5 个 Chunk，过滤相似度低于 0.3 的结果
3. **构建提示**：将检索到的 Chunk 内容拼接到 System Prompt 中，包含防幻觉规则和相关性判断指令
4. **LLM 生成**：将 System Prompt + 对话历史（最近10轮） + 用户问题发给大模型，生成最终回答

### 3.2 关键设计说明

**知识库 → 文档 → 切片**：三层层级结构

- 一个知识库可以包含多个文档
- 一个文档被切成多个切片
- 检索时以切片为单位匹配

**知识库配置字段**（新增）：

- `embeddingModel`：该知识库使用的 Embedding 模型
- `chunkSize`：分块大小（默认 500）
- `chunkOverlap`：分块重叠（默认 50）

**文档状态机**：

```
PENDING → PARSING → CHUNKING → EMBEDDING → COMPLETED
    │         │          │           │
    ▼         ▼          ▼           ▼
  (等待)   PARSE_FAILED CHUNK_FAILED EMBEDDING_FAILED
```

**FileStorage 引用计数**：

- 多个文档可能引用同一个文件（通过 MD5 去重）
- `refCount` 记录被引用次数，为 0 时才真正删除文件

**会话与知识库的动态关系**：

- `Conversation.kbIds` 是 JSON 数组，存储关联的知识库 ID
- 发送消息时，**优先使用请求中的 kbIds**（前端可动态切换）
- 前端 x 掉知识库后传空数组 → 不走 RAG
- kbIds 变更会同步更新到会话记录

## 四、向量检索原理

### 4.1 什么是 Embedding

Embedding（嵌入）是将文本转换为固定维度的浮点数向量的技术。

```
"退货需要什么条件"  ──Embedding──▶  [0.023, -0.156, 0.891, ..., 0.034]
                                        （1536 个浮点数）
```

**核心特性**：语义相近的文本，向量也相近。

```
"退货条件"  →  [0.12, 0.85, -0.33, ...]
"退款政策"  →  [0.11, 0.82, -0.30, ...]   ← 与"退货条件"很接近
"今天天气"  →  [-0.56, 0.12, 0.77, ...]   ← 完全不同的方向
```

### 4.2 余弦相似度

衡量两个向量方向的一致性，范围 [-1, 1]，值越大越相似。

```
                    cos(θ) = (A · B) / (|A| × |B|)

其中：
  A · B   = Σ(ai × bi)     点积
  |A|     = √Σ(ai²)        向量模长
  |B|     = √Σ(bi²)        向量模长
```

直觉理解：
- `cos = 1.0`：两个向量完全同向（语义相同）
- `cos = 0.0`：两个向量正交（语义无关）
- `cos = -1.0`：两个向量反向（语义相反）

### 4.3 检索过程

```
1. 用户问题向量化
   "退货条件" → queryVector = [0.12, 0.85, -0.33, ...]

2. 遍历所有 Chunk 向量，计算余弦相似度
   chunk_001: cos(query, chunk_001) = 0.92  ← 高度相关
   chunk_002: cos(query, chunk_002) = 0.85  ← 相关
   chunk_003: cos(query, chunk_003) = 0.21  ← 不相关（低于阈值0.3，被过滤）
   chunk_004: cos(query, chunk_004) = 0.78  ← 较相关
   ...

3. 过滤：相似度 ≥ 0.3 的结果才保留

4. 按相似度排序，取 Top-5
   [chunk_001(0.92), chunk_002(0.85), chunk_004(0.78), ...]

5. 将 Top-5 的原文内容拼接到 System Prompt
```

### 4.4 本项目的实现选择

| 设计点 | 生产方案 | 本项目选择 | 原因 |
|--------|---------|-----------|------|
| 向量存储 | Milvus/Weaviate | ConcurrentHashMap + 启动重建 | 教学简化，无需额外部署 |
| 检索算法 | ANN（近似最近邻） | 暴力遍历 | 数据量小，精确度高 |
| 相似度计算 | 向量数据库内置 | 手写余弦相似度 | 学生可以看到完整实现 |
| 相似度阈值 | 可调 | 默认 0.3 | 过滤不相关结果，日志可见过滤过程 |

### 5.3 核心数据流向

```
前端上传文档
    │
    ▼
DocumentController.uploadDocument()
    │
    ▼
DocumentService.uploadDocument()
    ├── 文件落盘 → FileStorage
    ├── 创建 Document 记录
    └── 事务提交后异步调用 DocumentProcessingService.processDocumentAsync()
                │
                ▼
        ┌─ 阶段1：DocParserClient.parseDocument()
        │       └── 调用 Python doc-parser /parse
        │           （不可用时直接报错，无降级）
        │
        ├─ 阶段2：createChunks(doc, parseResult, kb.chunkSize, kb.chunkOverlap)
        │       ├── 优先使用 doc-parser 的分块
        │       └── 否则 simpleChunking（使用知识库配置的大小和重叠）
        │
        └─ 阶段3：embedChunks(kbId, chunks, kb.embeddingModel)
                ├── EmbeddingService.embedBatch(contents, embeddingModel)
                └── VectorStoreService.upsertBatch()

前端发送消息
    │
    ▼
ChatController → ChatService.sendMessage()
    ├── 确定知识库：request.kbIds > conversation.kbIds（空数组=不走RAG）
    ├── 保存用户消息
    ├── RetrievalService.search()
    │       ├── EmbeddingService.embed(query)
    │       ├── VectorStoreService.search(kbIds, queryVector, topK=20)
    │       └── filterAndLimit(threshold=0.3, topK=5)
    │            └── 日志打印: 过滤前分数范围 → 过滤后数量
    ├── buildHistoryMessages(最近10轮=20条消息)
    ├── LLMService.generateRAGResponse(query, results, history)
    │       └── buildRAGSystemPrompt（防幻觉 + 相关性判断 + 来源标注）
    └── 保存 AI 消息 + 引用信息（含 content + score + docName）

应用启动
    │
    ▼
VectorIndexRebuilder.rebuildVectorIndex()
    ├── 遍历所有知识库
    ├── 查询 embeddingStatus=2（已向量化）的 Chunk
    ├── 批量调用 EmbeddingService.embedBatch() 重新生成向量
    └── 写入 VectorStoreService 内存
```
