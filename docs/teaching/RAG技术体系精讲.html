1 为什么需要 RAG？

大模型的三个硬伤与 RAG 的诞生

大模型很强，但有三道"硬伤"让它无法胜任企业级知识问答。本章从这三个痛点出发，带你理解 RAG 诞生的必然性，以及它如何成为当下最主流的解决方案。

LLM 的三个硬伤
  • 知识截止（Knowledge Cutoff）：训练数据有截止日期，比如 GPT-4o 的训练数据截止到2024年6月，你问它2025年底的事它不知道
  • 私有数据盲区：它读过维基百科但没读过你公司的内部文档、产品手册、操作规范
  • 幻觉问题（Hallucination）：不知道就编。比喻："考试不会也不留白，硬写一个看起来像答案的东西"
⭐ 幻觉是 LLM 最危险的问题——因为它编的答案看起来特别自信、特别合理
传统解决方案的困境
方案思路问题
微调(Fine-tuning)用私有数据重新训练模型成本高（几千到几万美元）、更新慢（每次数据变化都要重训）、可能遗忘原有能力
全量塞上下文把所有文档塞进 PromptToken 有限（128K也塞不下1000页）、成本随输入长度线性增长、"Lost in the Middle"注意力衰减
传统搜索+人工先搜索再人工整理效率低、无法自动生成答案、依赖人工
RAG 的诞生

2020年，Facebook AI Research (现 Meta AI) 发表论文《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》

与其让模型记住所有知识，不如让它学会"查资料"。就像开卷考试——先翻书找到相关内容，再根据书上的内容回答问题。

为什么 RAG 迅速成为主流：更新知识只需更新文档库（不需要重训模型）、成本低、可追溯来源、减少幻觉

闭卷 vs 开卷考试 — RAG 核心直觉
⭐ 一句话记住 RAG：让 AI 先查资料再答题，而不是凭记忆硬编
⭐ 截至2026年，RAG 是大模型落地最主流的技术方案，几乎所有企业级 AI 应用都在用
RAG vs Fine-tuning vs 长上下文
维度RAGFine-tuning长上下文
知识更新即时（更新文档库）慢（需重训）即时（但每次都要传）
成本低（只需检索+生成）高（训练成本）高（Token费用）
幻觉控制好（有来源约束）一般一般
可追溯是（标注来源）否否
适用场景知识密集型问答风格/格式适配单次长文档分析
本章要点
  • LLM 有知识截止、私有数据盲区、幻觉三大硬伤
  • 微调成本高、全量塞上下文受 Token 限制、传统搜索效率低
  • RAG 核心直觉：让 AI 先"查资料"再"答题"
  • RAG 在知识更新、成本、幻觉控制、可追溯性上优于微调和长上下文
  • RAG 是2026年企业级 AI 应用最主流的技术方案

2 RAG 的全景架构

两条管道、六个环节

RAG 不是一条直线，而是两条并行的管道：一条负责"备课"（把文档变成可检索的向量），一条负责"答题"（用户提问时检索并生成）。理解这两条管道，就掌握了 RAG 的骨架。

RAG = 让 AI 先"查资料"再"答题"，查到的资料就是它回答的依据
两条管道
RAG 两条管道完整流程
⭐ 离线管道和在线管道的 Embedding 必须用同一个模型，否则向量不在同一个空间，检索结果无意义
六个核心环节

本章要点

  • 离线管道：文档→解析→分块→向量化→入库
  • 在线管道：提问→向量化→检索→Prompt组装→LLM生成
  • 查询和文档必须用同一个 Embedding 模型

3 文档解析 🟢 基础

把 PDF、Word 变成机器可读的纯文本

RAG 的第一步是让机器"读懂"你的文档。PDF、Word 这些人类熟悉的格式，对机器来说却是复杂的二进制结构。本章带你理解解析的必要性，以及如何选对工具。

为什么不能直接用文件？
🎯 通俗版：你以为 PDF 里存的是文字？不，它存的是"在坐标(100,200)处画一个12号宋体的「退」字"这样的指令。就像你给朋友发了一张照片，他看到的是一张图，但电脑里存的是一串像素数据。

技术版：PDF 的内部是对象流（Object Streams）和内容流（Content Streams），存储的是绘制指令（如 Tj 显示文本、m 移动、l 画线），不是纯文本。Word (.docx) 是一个 ZIP 压缩包，解压后是 XML 结构。图片里的文字需要 OCR（光学字符识别）才能提取。

解析质量对 RAG 的影响

解析是 RAG 链条的第一环，遵循 Garbage In, Garbage Out 原则：解析错了，后面的分块、向量化、检索、生成全都会错。表格行列错乱、多列阅读顺序颠倒、页眉页脚混入正文，都会导致检索到无关内容或遗漏关键信息。

⭐ 解析质量往往被忽视，但它是 RAG 效果的上限。复杂文档建议用 Unstructured 或 LlamaParse 做布局感知解析。
解析的核心挑战

表格结构

行列关系容易丢失

多列排版

阅读顺序可能错乱

页眉页脚

需要过滤掉噪音

特殊内容

公式、图表、水印

  • 解析面临表格、多列、页眉页脚、公式等挑战
  • 根据场景选工具：简单用 PyMuPDF，复杂用 Unstructured
  • 图片文字需 OCR

4 文本分块 🟢 基础

把长文拆成可检索的段落

一本 500 页的书不能整本丢给 LLM，也不能整本去做向量检索。分块就是把长文本切成可管理的片段，同时保证每块语义完整、边界不割裂。本章讲透分块策略与重叠的妙用。

为什么必须分块？
  • • Token 限制：LLM 有上下文窗口限制，不能把整本书塞进去
  • • 检索精度：整篇文档匹配太粗糙，段落级匹配才精准
  • • 成本控制：只发相关段落给 LLM，减少 Token 消耗
一本500页的书，你不会整本丢给别人让他找答案。你会先拆成一章一章、一段一段，告诉他"答案大概在第三章第二段"
策略1：固定窗口分块（最基础）
🎯 通俗版：就像用尺子量着切香肠，每段 500 字，但相邻两段故意重叠 50 字。这样"第 498 到 502 字"那句完整的话，不会因为正好在边界被切成两半。

技术版：参数 chunk_size（每块大小）+ overlap（重叠区域）。重叠区域确保跨块边界的句子不会被硬截断，检索时若命中重叠区，相邻块都能提供上下文。

固定窗口分块与重叠示意
策略2：句子边界优化

不在字的中间切，尽量在句号或换行处截断。

Python
# 在句子边界处截断 if end < len(text): last_period = chunk.rfind('。') last_newline = chunk.rfind('\n') break_point = max(last_period, last_newline) if break_point > chunk_size // 2: chunk = chunk[:break_point + 1]
策略3 & 4

语义分块：按文档自然结构（标题、段落、列表）切分。优点：语义完整；缺点：块大小不均匀。

递归分块（LangChain 默认）：先按大分隔符（\n\n）切，太大再按小分隔符（\n）切，还太大按句号切。层层递进，保证每块不超过上限。

Python（LangChain 递归分块 — 实际项目最常用）
from langchain.text_splitter import RecursiveCharacterTextSplitter splitter = RecursiveCharacterTextSplitter( chunk_size=500, # 每块最多 500 字符 chunk_overlap=50, # 相邻块重叠 50 字符 separators=["\n\n", "\n", "。", "！", "？", "；", " ", ""] # 优先按段落切 → 按行切 → 按句号切 → 逐字切 ) chunks = splitter.split_text(long_document_text) # chunks = ["第一块内容...", "第二块内容...", ...]
Java（Spring Boot 手动实现递归分块）
public List<String> recursiveSplit(String text, int chunkSize, int overlap) { String[] separators = {"\n\n", "\n", "。", " "}; return doSplit(text, separators, 0, chunkSize, overlap); } private List<String> doSplit(String text, String[] seps, int level, int size, int overlap) { if (text.length() <= size) return List.of(text); String sep = level < seps.length ? seps[level] : ""; String[] parts = text.split(Pattern.quote(sep)); List<String> chunks = new ArrayList<>(); StringBuilder buf = new StringBuilder(); for (String part : parts) { if (buf.length() + part.length() > size && buf.length() > 0) { chunks.add(buf.toString()); // 保留重叠区 buf = new StringBuilder(buf.substring(Math.max(0, buf.length() - overlap))); } buf.append(part).append(sep); } if (buf.length() > 0) chunks.add(buf.toString()); return chunks; }
分块大小的影响

块太大

检索定位不精准，"大海捞针"，返回大段不相关内容

块太小

丢失上下文，"只见树木不见森林"，一句话被切成两半

⭐ 经验值：200-1000 字，重叠取块大小的 10%-20%。具体要根据文档类型和检索效果调整
⭐ Token ≠ 字。LLM 用 BPE 算法切"子词"：中文 1 字 ≈ 1.5 token，英文 1 词 ≈ 1-2 token。所以 chunk_size 用"字数"算时，要考虑 token 换算
本章要点
  • 分块为了 Token 限制、检索精度、成本控制
  • 固定窗口 + 重叠是最基础策略
  • 尽量在句子边界截断
  • 块太大检索不准，块太小丢上下文
  • 经验值：200-1000字，重叠10%-20%

5 向量化（Embedding） 🟡 进阶

RAG 最核心的一章 — 从字符串匹配到语义理解

Embedding 是 RAG 的灵魂：它让机器从"字面匹配"升级到"语义理解"。搜"退货"能命中"退款流程"，靠的就是向量化把意思相近的文本映射到空间中的相近位置。本章从直觉到原理，讲透向量化。

从字符串匹配到语义理解

传统搜索（关键词匹配）

搜"退货"找不到写着"退款流程"的文档

向量搜索（语义匹配）

搜"退货"能找到"退款流程"，因为它们意思接近

Embedding 让机器第一次能理解"意思"——不是比较字面是否相同，而是判断含义是否接近
Embedding 的直觉理解
🎯 通俗版：给每段文字发一个 GPS 坐标，不同的是这个坐标在 1536 维空间里。意思相近的话，坐标就靠在一起；意思差得远，坐标就离得远。

技术版：Embedding 将变长文本映射为固定维度的稠密向量，语义相似的文本在欧氏空间或余弦空间中距离更近。

举例："退货政策" 和 "退款流程" → 坐标很近；"退货政策" 和 "今天天气" → 坐标很远

向量空间中的语义聚类
"退货需要满足什么条件？" │ │ Embedding Model ▼ [0.023, -0.156, 0.891, 0.045, ..., 0.034] ← 1536 个浮点数
技术演进时间线
One-Hot（最早） ↓ Word2Vec（2013，Google） ↓ GloVe（2014） ↓ ELMo（2018） ↓ BERT（2018） ↓ 专用Embedding（2023~至今）

One-Hot：每个词一个维度，维度爆炸且不含语义。Word2Vec（2013，Google）革命性突破：基于分布式假设——"一个词的含义由它周围的词决定"，通过 Skip-gram 或 CBOW 预测上下文词来学习词向量，使语义相近的词（如"国王"与"女王"）在向量空间中自然靠近。BERT：Transformer 架构，双向上下文。专用模型：text-embedding-3, BGE, Jina 等，专门为检索优化。

Embedding 模型的训练原理

核心思想：对比学习（Contrastive Learning）

🎯 通俗版：就像教小孩认人——把"爸爸"和"妈妈"的照片放一起（正样本），把"爸爸"和"汽车"的照片放一起（负样本），反复训练后，模型学会把"爸爸""妈妈"归到相近的位置，把"汽车"归到远处。

技术版：通过正负样本对优化损失函数，使正样本对的向量距离减小、负样本对的向量距离增大。训练数据：问答对、段落对、翻译对等。

  • 正样本（语义相似的文本对）→ 训练模型让它们的向量靠近
  • 负样本（语义不相关的文本对）→ 训练模型让它们的向量远离
模型学到的不是"这个词什么意思"，而是"什么样的文本对应该靠近"
维度的含义

维度 = 信息容量。用 2 个数只能表示平面上的点，用 3 个能表示空间中的点，用 1536 个能编码极丰富的语义信息。

常见维度：384（小模型）、768（中等）、1536（OpenAI默认）、3072（大模型）。维度越高 → 信息越丰富，但计算成本越大。

常见 Embedding 模型对比
模型提供商维度特点
text-embedding-3-smallOpenAI1536性价比高，英文强
text-embedding-3-largeOpenAI3072最高精度，成本高
BGE-M3BAAI(智源)1024开源，中文强，支持多语言

⭐ 查询和文档必须用同一个 Embedding 模型！否则向量不在同一个空间，算出来的相似度没有意义
⭐ Embedding 不是万能的。反义词（"好"和"不好"）的向量可能很近，因为它们经常出现在相似的上下文中

本章要点
  • Embedding 让机器理解语义而非字面匹配
  • 语义相近的文本向量在空间中靠近
  • 训练原理：对比学习，正负样本对

  • 维度越高信息越丰富，成本也越大
  • 反义词可能向量接近（上下文相似）

6 相似度计算与向量检索 🟡 进阶

余弦相似度、ANN、向量数据库

有了向量，如何快速找到"最相似"的那几条？本章讲相似度度量（为什么用余弦）、近似最近邻算法（为什么暴力搜索不够），以及向量数据库的选型。

余弦相似度 (Cosine Similarity)
🎯 通俗版：两支箭射出去——看的不是飞了多远，而是方向是否一致。方向越一致，cos(θ) 越接近 1；方向完全相反，cos(θ) 接近 -1。

技术版：余弦相似度 = 两向量夹角的余弦值，归一化内积，与向量模长无关，适合比较不同长度的文本向量。

余弦相似度几何直观
cos(θ) = A·B|A| × |B| = Σ(aᵢ×bᵢ)√Σaᵢ² × √Σbᵢ²

范围 [-1, 1]，1=完全一致，0=无关，-1=完全相反。为什么适合文本：不受向量长度（文本长度）影响。

欧氏距离的陷阱 — 长度差异导致距离虚高
内积的偏见 — 长向量天然"占便宜"
度量方式直觉公式核心优点缺点RAG 推荐
余弦相似度 量角度 A·B / (|A|×|B|) 不受文本长短影响 — 首选 ✓
欧氏距离 量尺子 √Σ(aᵢ-bᵢ)² 直观好理解 受向量长度影响，长短文本距离天然大 少用
内积 方向+大小 Σ(aᵢ×bᵢ) 归一化后等价余弦 未归一化时受模长干扰 归一化后可用
同一次查询 · 三种度量 · 三种结果 — 余弦相似度是最可靠的选择
⭐ 实际工程中大多数向量数据库默认用余弦相似度，因为 Embedding 模型输出的向量长度不统一，余弦天然免疫这个问题
代码实现
Python
import numpy as np def cosine_similarity(a, b): dot_product = np.dot(a, b) norm_a = np.linalg.norm(a) norm_b = np.linalg.norm(b) if norm_a == 0 or norm_b == 0: return 0.0 return dot_product / (norm_a * norm_b)
Java
// Java 实现余弦相似度 public static double cosineSimilarity(float[] a, float[] b) { if (a.length != b.length) return 0.0; double dot = 0, normA = 0, normB = 0; for (int i = 0; i < a.length; i++) { dot += a[i] * b[i]; normA += a[i] * a[i]; normB += b[i] * b[i]; } if (normA == 0 || normB == 0) return 0.0; return dot / (Math.sqrt(normA) * Math.sqrt(normB)); }
暴力搜索 vs ANN

暴力搜索：遍历所有向量计算相似度，O(n×d)，精确但慢。

ANN（近似最近邻）：牺牲少量精度换大量速度。

🎯 通俗版：HNSW 像社交网络——每个人认识一些"朋友"，找人的时候不是挨个问，而是"我朋友的朋友里有没有他？"层层跳转，很快就能定位。

HNSW 技术原理：多层图结构，顶层稀疏（快速定位大致区域），底层稠密（精确搜索）。从顶层入口开始，逐层向下，每层贪心搜索到更近的邻居。层数约 log n，每层访问节点数有限，故整体复杂度约 O(log n)。

IVF 直觉：先把向量分到若干"区域"（聚类），查询时只搜离查询向量最近的几个区域，大幅减少计算量。

阈值过滤

不是所有"最近的"都是"相关的"。0.3 阈值的经验值——低于此分数视为不相关，过滤掉。

⭐ 阈值需根据实际数据调优，可在日志中打印分数范围观察分布

数据库类型特点

Qdrant开源/自部署Rust写的，性能好

  • 暴力搜索精确但慢，ANN 快但近似
  • HNSW、IVF 是主流 ANN 索引
  • 阈值过滤（如0.3）去除噪声
  • 向量数据库按规模选型

7 Prompt Engineering 🟢 基础

把检索结果注入提示词，约束 LLM 输出

检索到的内容不会自动变成答案，需要通过 Prompt 明确告诉 LLM：只基于这些参考内容回答，不要编造。本章讲 Prompt 的组装结构、防幻觉策略，以及 Temperature 等参数对 RAG 的影响。

Prompt 在 RAG 中的角色
你招了一个超级聪明的实习生（LLM），Prompt 就是你给他的岗位说明书+工作任务单

System Prompt = 岗位说明书（全程生效）。User Prompt = 具体任务（每次不同）。

RAG Prompt 的核心构成
Prompt 各层组装顺序
System Prompt 模板
你是基于参考内容回答问题的助手。 【最高优先级】绝对禁止幻觉。只使用以下参考内容回答，不得编造。 若参考内容与问题不相关，回答"❌ 未找到相关信息"。 回答时标注来源，格式：【来源：文档名】 参考内容： --- {retrieved_chunks} --- 用户问题：{user_question} 再次提醒：只使用参考内容，不要编造。
防幻觉的三重保障

弱 Prompt 容易导致 LLM 凭记忆"补充"或编造细节。对比：

❌ 弱 Prompt（易幻觉）

"根据以下内容回答：{chunks}。问题：{q}" — 没有明确禁止编造，LLM 可能自由发挥

✅ 强 Prompt（防幻觉）

开头声明"绝对禁止幻觉"、要求标注来源、结尾再次提醒"只使用参考内容" — 三重约束显著降低编造

  • 开头声明核心原则
  • 用真实文档名做来源标注示例
  • 结尾再次提醒
Temperature 与其他参数

Temperature：创造性旋钮。0-0.2：严谨会计（确定性高）。0.5-0.8：文案写手（有创意不离谱）。1.0+：喝醉的诗人（可能胡言乱语）。

Top-P：累积概率截断。Max Tokens：输出长度限制。RAG 场景推荐：Temperature 0.1-0.3，保证忠于参考内容。

⭐ Lost in the Middle 效应：LLM 对开头和结尾的内容关注更多，中间容易被忽略。实践技巧：把最重要的参考内容放在 Prompt 的开头和结尾位置
本章要点
  • System Prompt 设定角色和约束，User Prompt 传具体问题
  • 防幻觉：开头声明+来源标注+结尾提醒
  • RAG 场景 Temperature 宜低（0.1-0.3）
  • Lost in the Middle：重要内容放开头和结尾

8 RAG 的进阶优化 🔴 深入

Rerank、混合检索、Query 改写、Agentic RAG

基础 RAG 能跑起来，但效果往往有提升空间。Rerank 精排、混合检索、Query 改写、Agentic RAG 是业界常用的进阶手段。本章逐一拆解，帮你从"能用"走向"好用"。

Rerank（重排序）
🎯 通俗版：渔网捞鱼——先撒大网（向量检索）捞 20 条，再让老渔夫（Rerank）逐条挑出值钱的。先求快（能不能找到），再求准（是不是你要的）。

技术原理：双塔模型分别编码 query 和 doc，可预计算 doc 向量，检索快但粗；交叉编码器把 query+doc 拼一起输入，逐对打分，慢但准。典型流程：向量检索 Top-20 → Rerank → Top-5。

Rerank 两阶段流程

为什么不直接用交叉编码器搜全库？交叉编码器需要把 query 和每条文档拼在一起逐一打分，100 万条文档就要跑 100 万次推理，太慢了。所以先用向量检索（毫秒级）从 100 万条里粗筛 20 条，再让交叉编码器精排这 20 条（几百毫秒），兼顾速度和精度。

Rerank 模型/服务类型特点适用场景
Cohere Rerank云 API开箱即用，效果好，按调用付费快速集成、商业项目
BGE-reranker-v2-m3开源模型智源出品，中文效果强，可本地部署中文场景、私有化部署
cross-encoder/ms-marco-*开源模型HuggingFace 上最经典的交叉编码器系列英文场景、学习入门
Jina Reranker v2开源/API多语言支持，长文本友好多语言、长文档
Python（完整流程：向量检索 → Rerank → Top-5）
from sentence_transformers import CrossEncoder # 1. 向量检索粗筛（假设已有向量库） candidates = vector_db.search(query_embedding, top_k=20) # candidates = [{"text": "退货需在7天内...", "score": 0.82}, ...] # 2. 加载交叉编码器 Rerank 模型 reranker = CrossEncoder("BAAI/bge-reranker-v2-m3") # 3. 逐对打分：把 (query, doc) 配对输入 pairs = [(query, doc["text"]) for doc in candidates] scores = reranker.predict(pairs) # 4. 按 Rerank 分数重新排序，取 Top-5 for i, doc in enumerate(candidates): doc["rerank_score"] = scores[i] reranked = sorted(candidates, key=lambda x: x["rerank_score"], reverse=True)[:5] # reranked 就是精排后的 Top-5，注入 Prompt

混合检索（Hybrid Search）

向量检索擅长语义匹配（"退货"→"退款流程"），但碰到专有名词、型号、代码就不行了——搜"iPhone 15 Pro Max"，向量可能匹配到一堆手机评测，却漏掉精确包含这个词的那条文档。BM25 恰好相反：它按关键词匹配，精确但不懂同义词。两者互补，混合使用效果最好。

向量 + BM25 混合检索
BM25 是什么？
🎯 通俗版：BM25 就是"高级版 Ctrl+F"。你搜"退货政策"，它在所有文档里数哪些文档包含"退货"和"政策"这两个词，包含得越多、越集中，分数越高。但它比普通搜索聪明一点：一个词出现 3 次和 30 次的差距没那么大（词频饱和），而且在一篇短文档里出现 3 次比在一篇万字长文里出现 3 次更有意义（长度归一化）。

技术版：BM25 = TF-IDF 的实用改进版

  • TF（词频）：这个词在文档里出现了几次？出现越多，文档越可能相关
  • IDF（逆文档频率）：这个词在所有文档中有多稀有？"退货"只在 5% 的文档里出现 → 区分度高，IDF 大；"的"在 99% 文档里出现 → 没区分度，IDF 极小
  • BM25 的改进：加了词频饱和（出现 3 次 vs 30 次差距缩小，参数 k₁ 控制）和文档长度归一化（长文档不占便宜，参数 b 控制）
BM25(q, d) = Σ IDF(qᵢ) × f(qᵢ, d) × (k₁ + 1)f(qᵢ, d) + k₁ × (1 - b + b × |d|/avgdl)

k₁ 通常取 1.2-2.0，b 通常取 0.75。不需要背公式，理解"词频饱和 + 长度归一化"即可。

⭐ BM25 不需要训练，不需要 GPU，不需要 Embedding 模型——它是纯数学公式，对每条文档计算一个分数，速度极快。这也是为什么它适合作为向量检索的补充。

向量检索擅长

"退货" 能搜到 "退款流程"
"how to return" 能搜到 "退货政策"
同义词、跨语言、模糊语义

BM25 擅长

"iPhone 15 Pro Max" 精确匹配
"ERROR_CODE_4012" 精确匹配
专有名词、型号、代码、ID

RRF 融合：两路结果怎么合并？
🎯 通俗版：你让两个评委给 10 篇作文排名。评委 A 的第 1 名和评委 B 的第 3 名，综合来看应该排在前面。RRF 就是把两个排名表合并成一个的方法——只看排名位置，不看原始分数（因为两路分数量纲不同，没法直接比）。

RRF（Reciprocal Rank Fusion）倒数排名融合

RRF_score(d) = 1k + rank向量(d) + 1k + rankBM25(d)

k 是平滑常数（通常 = 60），防止排名第 1 的文档权重过大。

举个具体例子：

文档向量检索排名BM25 排名RRF 得分最终排序
退货政策.pdf第 1 名第 2 名1/61 + 1/62 = 0.0325🥇 第 1
退款流程.pdf第 2 名第 5 名1/62 + 1/65 = 0.0315🥈 第 2
售后FAQ.pdf第 8 名第 1 名1/68 + 1/61 = 0.0311🥉 第 3
商品介绍.pdf第 3 名未召回1/63 + 0 = 0.0159第 4

可以看到：退货政策在两路都排前列，RRF 分数最高。售后FAQ 虽然向量排第 8，但 BM25 排第 1（精确命中关键词），融合后提到第 3。商品介绍只被向量检索召回，分数减半。这就是混合检索的威力——两路互补，既不漏掉语义相关的，也不漏掉精确匹配的。

Python（BM25 + 向量检索 + RRF 融合完整实现）
from rank_bm25 import BM25Okapi import jieba # ===== 1. 构建 BM25 索引（离线，只需一次）===== documents = ["退货需在7天内申请...", "退款将在3个工作日到账...", ...] # 中文需要先分词 tokenized_docs = [list(jieba.cut(doc)) for doc in documents] bm25 = BM25Okapi(tokenized_docs) # ===== 2. 在线检索：两路并行 ===== query = "iPhone 15 退货政策" # 路线 A：向量检索 Top-20 vector_results = vector_db.search(embed(query), top_k=20) # 路线 B：BM25 检索 Top-20 query_tokens = list(jieba.cut(query)) bm25_scores = bm25.get_scores(query_tokens) bm25_top = sorted(enumerate(bm25_scores), key=lambda x: x[1], reverse=True)[:20] # ===== 3. RRF 融合 ===== k = 60 rrf_scores = {} for rank, doc in enumerate(vector_results): rrf_scores[doc.id] = rrf_scores.get(doc.id, 0) + 1 / (k + rank + 1) for rank, (doc_idx, score) in enumerate(bm25_top): rrf_scores[doc_idx] = rrf_scores.get(doc_idx, 0) + 1 / (k + rank + 1) # 按 RRF 分数排序，取 Top-5 final = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)[:5]
⭐ 实际项目中，Milvus 2.4+ 和 Elasticsearch 8.x 已内置混合检索功能，不需要手动实现 RRF——一个 API 调用即可同时跑向量+关键词检索并自动融合。
Query 改写：为什么用户的原话去检索效果差？
🎯 通俗版：你去图书馆找书，你说"那个讲怎么退东西的书在哪？"，图书管理员一脸懵。但如果你说"请问《电商售后服务管理规范》在哪？"，管理员秒定位。Query 改写就是在检索之前，先把用户的"口语化提问"翻译成更接近"文档写法"的版本。

核心问题：用户的提问方式和文档的书写方式往往差异很大。直接拿用户原话去做向量检索，召回的文档可能不够精准。改写后再检索，效果显著提升。

改写策略①：多轮指代消解

场景：用户在多轮对话中说了"那它呢？""换货呢？"这种追问，直接拿去检索什么都搜不到——向量库里没有任何文档匹配"那它呢"。

做法：用 LLM 结合对话历史，把模糊的追问还原成一句完整的问题，再拿这个完整问题去检索。

阶段内容
用户第 1 轮"iPhone 15 怎么退货？"
AI 回答"需在 7 天内申请，商品保持原包装..."
用户第 2 轮（原始）"那换货呢？" ← 直接检索 = 搜不到
LLM 改写后"iPhone 15 的换货政策是什么？" ← 这个才能精准检索
Python 实现
def rewrite_query(current_query, chat_history, llm): prompt = f"""根据对话历史，将用户最新问题改写为一句独立、完整的问题。 对话历史： {chat_history} 用户最新问题：{current_query} 改写后的完整问题：""" return llm.generate(prompt) # "那换货呢？" → "iPhone 15 的换货政策是什么？" rewritten = rewrite_query("那换货呢？", history, llm) results = vector_db.search(embed(rewritten), top_k=5)
改写策略②：HyDE — 用"假设答案"去检索
🎯 直觉：用户的问题是"退货流程是什么？"（6 个字），文档里写的是"消费者需在签收后 7 个自然日内提交退货申请，商品需保持原始包装..."（一大段专业描述）。6 个字和一大段话，向量距离可能不够近。

如果先让 LLM 根据问题"编"一段假设答案（哪怕不准确），这段假设答案在措辞上会比 6 个字的问题更接近真实文档 → 用假设答案的向量去检索 → 命中率更高。
检索方式用来做向量检索的文本与真实文档的距离
普通检索"退货流程是什么？"（6字）较远
HyDE"退货需在收货后7天内申请，商品保持原包装，联系客服获取退货地址..."（LLM 生成的假设答案）很近
Python 实现
def hyde_search(query, llm, vector_db): # 1. 让 LLM 先"编"一段假设答案 hypothetical = llm.generate(f"请用一段话简要回答：{query}") # 2. 用假设答案的向量（而非原问题）去检索 results = vector_db.search(embed(hypothetical), top_k=5) return results # 假设答案不需要准确，只需要措辞接近文档风格
改写策略③ & ④

Query Decomposition（问题分解）

用户问"向量检索和 BM25 的区别是什么？"——这是一个复合问题，单次检索很难同时找到两个主题的文档。拆成两个子问题分别检索，再合并结果：
→ "向量检索的原理是什么？"
→ "BM25 的原理是什么？"

Step-back Prompting（退一步提问）

用户问"为什么 BM25 在长文档上效果不好？"——直接检索可能找不到。先退一步问更宏观的问题"BM25 的计算原理是什么？"，检索到 BM25 的背景知识，LLM 再基于这些知识回答具体问题。
⭐ 这四种策略不是互斥的，可以组合使用。例如：先做指代消解 → 再做 HyDE → 再 Rerank。实际项目中根据效果选用。
Agentic RAG

传统 RAG：每次提问都检索一次。Agentic RAG：由 Agent 自主判断是否需要检索、检索几次、检索什么。

典型能力：判断问题是否需要查文档（闲聊可不检索）、多轮检索（第一次不够再查）、结合工具调用（查数据库、调 API）。适合复杂、多步骤的问答场景。

本章要点
  • Rerank：向量粗筛 Top-20 → 交叉编码器精排 Top-5，兼顾速度和精度
  • BM25：基于词频和逆文档频率的关键词检索，擅长精确匹配专有名词
  • 混合检索：向量（语义）+ BM25（关键词）互补，RRF 按排名融合
  • Query 改写：多轮指代消解、HyDE 假设答案、问题分解
  • Agentic RAG：Agent 自主判断是否检索、检索几次

9 RAG 的评估与未来 🟡 进阶

如何衡量 RAG 好坏，以及 RAG 会走向何方

RAG 没有标准答案，只有不断调优。本章介绍评估的四个维度、RAGAS 自动化评估框架，以及一份实用的调优清单，帮你系统化排查和提升 RAG 效果。

评估的四个维度
  • 检索质量：召回率、精确率、MRR
  • 生成质量：忠实度、相关性、完整性
  • 端到端：回答正确率
  • 用户体验：响应时间、满意度
RAGAS 评估框架

RAGAS（Retrieval Augmented Generation Assessment）用 LLM 作为评判器，无需人工标注即可自动化评估 RAG 各环节质量。

  • Faithfulness（忠实度）：回答是否完全基于检索到的上下文？有无编造？用 LLM 判断回答中的每个陈述是否能在上下文中找到支撑。
  • Answer Relevancy（回答相关性）：回答是否切题？用 LLM 生成"若用户问这个问题，理想回答应包含什么"，再与真实回答对比。
  • Context Precision（上下文精确率）：检索到的上下文中，有多少是真正有用的？有用片段排名越靠前，分数越高。
  • Context Recall（上下文召回率）：标准答案中的关键信息，有多少被检索到了？需要人工提供 ground truth 或由 LLM 生成。

实践建议：用 RAGAS 在测试集上跑基线，每次改分块/检索/Prompt 后复测，观察指标变化。

RAG 调优 Checklist
  • ☐ 解析：复杂文档是否用布局感知解析？表格、多列是否正确？
  • ☐ 分块：chunk_size、overlap 是否根据文档类型调过？是否在句子边界截断？
  • ☐ Embedding：查询和文档是否用同一模型？中文场景是否用中文优化模型？
  • ☐ 检索：Top-K、阈值是否合理？是否尝试 Rerank、混合检索？
  • ☐ Prompt：防幻觉指令是否足够强？Temperature 是否 ≤ 0.3？
  • ☐ 多轮：历史窗口是否足够？指代消解是否处理？

RAG 的未来

长上下文模型会取代 RAG 吗？答：不会，各有所长。长上下文适合单次长文档分析，RAG 适合动态知识库、多文档、可追溯。

趋势：多模态 RAG、端到端可训练的 RAG。

RAG 不是一个临时方案，而是 AI 系统"外接知识"的基础架构模式
本章要点
  • 评估：检索质量、生成质量、端到端、体验
  • RAGAS 提供自动化评估指标
  • 症状-原因-排查表帮助诊断
  • RAG 与长上下文互补，非替代
  • RAG 是外接知识的基础架构
🎯 三条核心洞察

1. RAG 的本质 — "检索+生成"的分工：检索负责找对资料，生成负责组织语言

2. 每个环节的质量都会影响最终效果 — 分块太粗检索不准，Prompt 太弱容易幻觉

3. 没有完美的 RAG — 只有不断调优的 RAG，这是一门实验科学
