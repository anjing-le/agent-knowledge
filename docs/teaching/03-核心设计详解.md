## 【核心1】文档解析：DocParserClient 提取文本

### 设计思路

RAG 的第一步是把原始文档（PDF、Word、TXT 等）变成纯文本。

本项目**必须依赖 doc-parser 服务**，没有降级方案。这确保教学时每个环节都是真实的外部调用。

调用流程：

```
DocumentProcessingService.parseDocument()
    │
    ├── docParserClient.isHealthy()  ← 检查 Python 服务是否可用
    │       │
    │       ├── 可用 → docParserClient.parseDocument(filePath, docType)
    │       │           └── HTTP POST /parse（multipart/form-data）
    │       │
    │       └── 不可用 → 直接返回错误（无降级）
    │                    "doc-parser 服务不可用，请确保已启动"
    │
    └── 返回 ParseResult { content, chunks, metadata }
```

### 关键代码

**DocParserClient** — 调用 Python 解析服务：

```java
// DocParserClient.java
public ParseResult parseDocument(String filePath, String docType) {
    String url = baseUrl + "/parse";

    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.MULTIPART_FORM_DATA);

    MultiValueMap<String, Object> body = new LinkedMultiValueMap<>();
    body.add("file", new FileSystemResource(new File(filePath)));
    body.add("doc_type", docType != null ? docType : "DOCUMENT_BASIC");

    HttpEntity<MultiValueMap<String, Object>> requestEntity =
            new HttpEntity<>(body, headers);

    ResponseEntity<String> response = restTemplate.exchange(
            url, HttpMethod.POST, requestEntity, String.class);

    return parseResponse(response.getBody());
}
```

**解析入口** — 无降级，doc-parser 不可用直接报错：

```java
// DocumentProcessingService.java
private DocParserClient.ParseResult parseDocument(Document doc) {
    String filePath = fileStorageRepository.findById(doc.getFileId())
            .map(fs -> fs.getStoragePath()).orElse(null);

    if (filePath == null) {
        return DocParserClient.ParseResult.error("文件存储路径不存在");
    }

    if (!docParserClient.isHealthy()) {
        return DocParserClient.ParseResult.error(
            "doc-parser 服务不可用，请确保 doc-parser 已启动（端口9001）");
    }

    return docParserClient.parseDocument(filePath, mapDocType(doc.getDocType()));
}
```

### 要点总结

| 设计点 | 说明 |
|--------|------|
| 服务拆分 | 解析逻辑放在独立的 Python 服务，Java 只做调用 |
| 健康检查 | 调用前先检查 `/health`，不可用时直接报错 |
| 无降级 | 不提供 Java 端简单提取作为 fallback，确保真实依赖 |
| 返回格式 | 统一返回 `ParseResult`（content + chunks + metadata） |

## 【核心2】智能分块：可配置的 Chunk 策略

### 设计思路

为什么要分块？

- **LLM 上下文窗口有限**：不能把整个文档塞给 LLM
- **检索精度**：整篇文档匹配不如段落级匹配准确
- **成本控制**：只把相关段落发给 LLM，减少 Token 消耗

分块参数**从知识库配置中读取**，前端可视化编辑：

```
createChunks(doc, parseResult, kb.getChunkSize(), kb.getChunkOverlap())
    │
    ├── doc-parser 返回了 chunks？
    │       │
    │       ├── 是 → 直接使用 doc-parser 的分块结果
    │       │         （按文档结构智能分块，如标题、段落、表格）
    │       │
    │       └── 否 → simpleChunking(doc, content, taskId, chunkSize, chunkOverlap)
    │                 （使用知识库配置的大小和重叠）
    │
    └── 返回 List<Chunk>
```

### 关键代码

**分块策略选择** — 参数来自知识库配置：

```java
// DocumentProcessingService.java
private List<Chunk> createChunks(Document doc, ParseResult parseResult,
                                  int chunkSize, int chunkOverlap) {
    List<Chunk> chunks = new ArrayList<>();
    String taskId = generateTaskId();

    // 优先使用 doc-parser 返回的分块结果
    if (parseResult.getChunks() != null && !parseResult.getChunks().isEmpty()) {
        for (ChunkData chunkData : parseResult.getChunks()) {
            Chunk chunk = new Chunk();
            chunk.setChunkId(generateChunkId());
            chunk.setDocId(doc.getDocId());
            chunk.setKbId(doc.getKbId());
            chunk.setContent(chunkData.getContent());
            chunk.setChunkIndex(chunkData.getIndex());
            chunk.setEmbeddingStatus(EmbeddingStatus.NOT_EMBEDDED.getCode());
            chunk.setCreatedAt(LocalDateTime.now());
            // ... 其他字段
            chunks.add(chunk);
        }
    } else if (parseResult.getContent() != null && !parseResult.getContent().isEmpty()) {
        // 使用知识库配置的分块参数
        chunks = simpleChunking(doc, parseResult.getContent(), taskId,
                                chunkSize, chunkOverlap);
    }

    return chunks;
}
```

**固定长度分块** — 参数可配置：

```java
// DocumentProcessingService.java
private List<Chunk> simpleChunking(Document doc, String content, String taskId,
                                    int chunkSize, int overlap) {
    List<Chunk> chunks = new ArrayList<>();
    log.info("[RAG] 分块参数: chunkSize={}, overlap={}, docId={}",
             chunkSize, overlap, doc.getDocId());

    int pos = 0;
    while (pos < content.length()) {
        int end = Math.min(pos + chunkSize, content.length());
        String chunkContent = content.substring(pos, end);

        // 尝试在句子边界处截断（句号或换行）
        if (end < content.length()) {
            int lastPeriod = chunkContent.lastIndexOf('。');
            int lastNewline = chunkContent.lastIndexOf('\n');
            int breakPoint = Math.max(lastPeriod, lastNewline);
            if (breakPoint > chunkSize / 2) {
                chunkContent = chunkContent.substring(0, breakPoint + 1);
                end = pos + breakPoint + 1;
            }
        }

        Chunk chunk = new Chunk();
        chunk.setContent(chunkContent.trim());
        chunk.setChunkIndex(index);
        chunk.setCreatedAt(LocalDateTime.now());
        // ... 设置其他字段
        chunks.add(chunk);

        pos = end - overlap;  // 重叠区域
    }

    return chunks;
}
```

### 分块配置可视化

知识库创建/编辑时，前端可以配置分块参数：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| Embedding 模型 | text-embedding-3-small | 该知识库使用的向量化模型 |
| 分块大小 | 500 | 每个 Chunk 的最大字数 |
| 重叠大小 | 50 | 相邻 Chunk 的重叠字数 |

知识库详情页顶部的 **RAG 配置栏** 实时显示这些参数。修改配置后，可以对文档点击「重新解析」使新配置生效。

│ AAAAAAAAAA BBBBBBBBBB CCCCCCCCCC DDDDDDDDDD EEEEEEEEEE ... │
└──────────────────────────────────────────────────────────────┘

分块结果（chunkSize=500, overlap=50）：

### 要点总结

| 设计点 | 说明 |
|--------|------|
| 参数可配置 | chunkSize 和 overlap 从知识库配置读取，前端可视化编辑 |
| 边界优化 | 尽量在句号或换行处截断，保持语义完整 |
| Token 估算 | 中文 1 字 ≈ 1.5 token，英文 1 字符 ≈ 0.25 token |
| 重新解析 | 修改配置后可对文档重新解析，使新参数生效 |

## 【核心3】向量化：调用 Embedding API 生成向量

### 设计思路

Embedding 的作用是将文本转换为数学向量，使得语义相近的文本在向量空间中距离也近。

Embedding 模型**从知识库配置读取**，默认使用 `text-embedding-3-small`，生成 1536 维的浮点数向量。

```
"退货需要满足什么条件？"
         │
         │  Embedding API（模型由知识库配置决定）
         ▼
[0.023, -0.156, 0.891, 0.045, ..., 0.034]   ← 1536 个浮点数
```

### 关键代码

**EmbeddingService** — 支持指定模型：

```java
// EmbeddingService.java
public List<List<Float>> embedBatch(List<String> texts, String embeddingModel) {
    // null 则用全局默认模型
    String actualModel = (embeddingModel != null && !embeddingModel.isBlank())
            ? embeddingModel : this.model;

    HttpHeaders headers = new HttpHeaders();
    headers.setContentType(MediaType.APPLICATION_JSON);
    headers.setBearerAuth(apiKey);

    Map<String, Object> body = new HashMap<>();
    body.put("model", actualModel);     // 使用知识库配置的模型
    body.put("input", texts);           // 支持批量输入
    body.put("dimensions", dimensions);

    HttpEntity<Map<String, Object>> request = new HttpEntity<>(body, headers);

    ResponseEntity<Map> response = restTemplate.exchange(
            apiUrl, HttpMethod.POST, request, Map.class);

    // 解析返回的 embedding 向量
    List<Map<String, Object>> data = (List) response.getBody().get("data");
    List<List<Float>> embeddings = new ArrayList<>();
    for (Map<String, Object> item : data) {
        List<Number> embedding = (List<Number>) item.get("embedding");
        List<Float> floatEmbedding = new ArrayList<>();
        for (Number num : embedding) {
            floatEmbedding.add(num.floatValue());
        }
        embeddings.add(floatEmbedding);
    }
    return embeddings;
}
```

**批量向量化** — 使用知识库配置的模型：

```java
// DocumentProcessingService.java — embedChunks 方法
private boolean embedChunks(String kbId, List<Chunk> chunks, String embeddingModel) {
    int batchSize = 20;  // 每批 20 个 chunk
    for (int i = 0; i < chunks.size(); i += batchSize) {
        List<Chunk> batch = chunks.subList(i, Math.min(i + batchSize, chunks.size()));

        List<String> contents = new ArrayList<>();
        List<String> chunkIds = new ArrayList<>();
        for (Chunk chunk : batch) {
            contents.add(chunk.getContent());
            chunkIds.add(chunk.getChunkId());
        }

        // 更新状态为向量化中
        chunkRepository.batchUpdateEmbeddingStatus(chunkIds, EmbeddingStatus.EMBEDDING.getCode());

        // 调用 Embedding API（使用知识库配置的模型）
        List<List<Float>> vectors = embeddingService.embedBatch(contents, embeddingModel);

        // 存储到向量库
        vectorStoreService.upsertBatch(kbId, chunkIds, vectors, contents);

        // 更新 chunk 的向量化状态
        for (Chunk chunk : batch) {
            chunk.setEmbeddingStatus(EmbeddingStatus.EMBEDDED.getCode());
            chunk.setVectorId(chunk.getChunkId());
        }
        chunkRepository.saveAll(batch);
    }
    return true;
}
```

### 要点总结

| 设计点 | 说明 |
|--------|------|
| 模型可配置 | 从知识库配置读取 embeddingModel，前端可选择 |
| 批量处理 | 每次发 20 个文本，减少 API 调用次数 |
| 接口标准 | 兼容 OpenAI Embedding API，可替换其他厂商 |
| 维度 | 1536 维，`text-embedding-3-small` 的默认维度 |
| 状态追踪 | 每个 Chunk 有 `embeddingStatus`（NOT_EMBEDDED → EMBEDDING → EMBEDDED） |

## 【核心4】向量检索：内存向量库 + 余弦相似度 + 阈值过滤

### 设计思路

向量检索的目标：给定用户问题的向量，在所有 Chunk 向量中找到最相似的 Top-K 个。

本项目使用内存实现（`ConcurrentHashMap`），适合教学和小规模数据。**重启后自动从数据库重建向量索引**。

存储结构：

```
store: ConcurrentHashMap<String, ConcurrentHashMap<String, VectorEntry>>
       ├── "kb_001" → { "chunk_001" → VectorEntry, "chunk_002" → VectorEntry, ... }
       ├── "kb_002" → { "chunk_005" → VectorEntry, "chunk_006" → VectorEntry, ... }
       └── ...

VectorEntry:
  ├── chunkId: "chunk_001"
  ├── vector: [0.023, -0.156, 0.891, ...]   ← 1536维
  └── content: "退货政策第一条原文..."
```

### 关键代码

**向量检索** — 遍历 + 余弦相似度：

```java
// VectorStoreService.java
public List<VectorSearchResult> search(List<String> kbIds,
                                        List<Float> queryVector, int topK) {
    List<VectorSearchResult> allResults = new ArrayList<>();

    for (String kbId : kbIds) {
        ConcurrentHashMap<String, VectorEntry> kbStore = store.get(kbId);
        if (kbStore == null || kbStore.isEmpty()) continue;

        for (VectorEntry entry : kbStore.values()) {
            float similarity = cosineSimilarity(queryVector, entry.getVector());
            allResults.add(new VectorSearchResult(
                    entry.getChunkId(), kbId, entry.getContent(), similarity));
        }
    }

    // 按相似度降序排列，取 Top-K
    allResults.sort((a, b) -> Float.compare(b.getScore(), a.getScore()));
    return allResults.stream().limit(topK).collect(Collectors.toList());
}
```

**余弦相似度** — 核心数学公式的实现：

```java
// VectorStoreService.java
private float cosineSimilarity(List<Float> a, List<Float> b) {
    if (a.size() != b.size()) return 0.0f;

    float dotProduct = 0.0f;   // 点积
    float normA = 0.0f;        // 向量 A 的模
    float normB = 0.0f;        // 向量 B 的模

    for (int i = 0; i < a.size(); i++) {
        dotProduct += a.get(i) * b.get(i);   // Σ(ai × bi)
        normA += a.get(i) * a.get(i);        // Σ(ai²)
        normB += b.get(i) * b.get(i);        // Σ(bi²)
    }

    if (normA == 0.0f || normB == 0.0f) return 0.0f;

    return dotProduct / (float) (Math.sqrt(normA) * Math.sqrt(normB));
    //     cos(θ) = (A·B) / (|A| × |B|)
}
```

### 启动时自动重建向量索引

// VectorIndexRebuilder.java
@EventListener(ApplicationReadyEvent.class)
public void rebuildVectorIndex() {
    log.info("[启动] 开始重建内存向量索引...");

    List<KnowledgeBase> kbList = knowledgeBaseRepository
            .findByIsDeletedFalseOrderByCreatedAtDesc();

    for (KnowledgeBase kb : kbList) {

        List<Chunk> chunks = chunkRepository

### 要点总结

| 设计点 | 说明 |
|--------|------|
| 存储结构 | 按知识库隔离，`kbId → chunkId → VectorEntry` |
| 检索方式 | 暴力遍历所有向量，计算余弦相似度 |
| 阈值过滤 | 默认 0.3，日志打印过滤前后的分数范围和数量 |
| 无降级 | Embedding 不可用时直接报错，不降级为关键词搜索 |
| 重启恢复 | 启动时自动从 DB 重建向量索引，无需手动重新解析 |
| 线程安全 | 使用 `ConcurrentHashMap`，支持并发读写 |

## 【核心5】RAG 生成：防幻觉 Prompt + 对话历史 + 来源标注

### 设计思路

RAG 的最后一步：将检索到的相关 Chunk 和用户问题一起发给 LLM，让 LLM 基于这些内容生成回答。

核心在于 **System Prompt 的构建** —— 包含防幻觉指令、相关性判断、来源标注规则、以及知识库参考内容。同时传入**对话历史**（最近10轮），实现多轮对话。

│ 2. 忠于原文                                      │

private List<Map<String, String>> buildHistoryMessages(String conversationId) {
    List<Message> messages = messageRepository
            .findByConversationIdOrderBySequenceAsc(conversationId);

    int maxMessages = 20; // 最近 10 轮 = 20 条消息
    int start = Math.max(0, messages.size() - maxMessages);

    List<Map<String, String>> history = new ArrayList<>();
    for (int i = start; i < messages.size(); i++) {

        history.add(Map.of("role", msg.getRole(), "content", msg.getContent()));

**ChatService — 完整的 RAG 问答流程（含动态知识库选择）**：

```java
// ChatService.java
public MessageResponse sendMessage(SendMessageRequest request) {
    Conversation conversation = ...;

    // 1. 保存用户消息
    Message userMessage = saveMessage(conversationId, "user", request.getContent());

    // 2. 确定知识库（前端可动态切换，空数组=不走RAG）
    List<String> kbIds;
    if (request.getKbIds() != null) {
        kbIds = request.getKbIds();        // 前端显式指定（包括空数组）
    } else if (conversation.getKbIds() != null) {
        kbIds = fromJsonList(conversation.getKbIds()); // 会话配置
    } else {
        kbIds = new ArrayList<>();         // 无知识库
    }
    conversation.setKbIds(kbIds.isEmpty() ? null : toJson(kbIds)); // 同步更新

    // 3. 执行知识检索（只有选了知识库才走 RAG）
    List<SearchResult> searchResults = new ArrayList<>();
    if (!kbIds.isEmpty()) {
        searchResults = retrieveKnowledge(request.getContent(), kbIds);
    }

    // 4. 加载对话历史 + 调用 LLM 生成回答
    String aiResponse = generateResponse(request.getContent(), searchResults, conversation);
    //  内部：buildHistoryMessages() + llmService.generateRAGResponse(msg, results, history)

    // 5. 保存 AI 消息 + 引用信息
    Message aiMessage = saveMessage(conversationId, "assistant", aiResponse);
    if (!searchResults.isEmpty()) {
        aiMessage.setReferences(toJson(searchResults));
        messageRepository.save(aiMessage);
    }

    return MessageResponse.fromEntity(aiMessage);
}
```

### 前端引用展示

AI 回答下方的**参考来源**区域展示：
- 文档名（加粗）
- 相似度（绿色百分比）
- 引用原文片段（灰色，最多150字截断）

### 要点总结

| 设计点 | 说明 |
|--------|------|
| 防幻觉 Prompt | 三重保障：核心原则 + 相关性判断 + 结尾再次提醒 |
| 来源标注 | 用真实文档名做示例，避免 LLM 输出占位符 |
| 对话历史 | 最近 10 轮（20条消息），存储在数据库，不因重启丢失 |
| 动态知识库 | 前端可实时选择/取消知识库，空数组=不走RAG |
| 引用展示 | 前端显示文档名 + 相似度 + 引用原文 |
| 相关性判断 | 即使有检索结果，主题不相关时 LLM 会回答 ❌ |

## 【核心6】启动重建：解决内存向量库重启丢失问题

### 设计思路

本项目使用 `ConcurrentHashMap` 作为内存向量库，重启后数据丢失。为此引入了 `VectorIndexRebuilder`，在应用启动后自动从数据库重建向量索引。

```
应用启动完成（ApplicationReadyEvent）
    │
    ▼
VectorIndexRebuilder.rebuildVectorIndex()
    │
    ├── 遍历所有未删除的知识库
    │
    ├── 对每个知识库，查询 embeddingStatus=2（已向量化）的 Chunk
    │
    ├── 按每批 20 条调用 EmbeddingService.embedBatch()
    │       └── 使用知识库配置的 embeddingModel
    │
    └── 写入 VectorStoreService.upsertBatch()
```

启动日志：
```
[启动] 开始重建内存向量索引...
[启动] 重建知识库向量: kbId=kb_xxx, name=xxx, chunkCount=52
[启动] 向量索引重建完成: 知识库=1个, 向量=52条
```

### 要点总结

| 设计点 | 说明 |
|--------|------|
| 触发时机 | `@EventListener(ApplicationReadyEvent.class)`，应用完全启动后 |
| 数据来源 | 数据库中 embeddingStatus=2 的 Chunk |
| 批量处理 | 每批 20 条调用 Embedding API |
| 模型一致 | 使用知识库配置的 embeddingModel，确保向量维度一致 |
| 容错 | 单批次失败不影响其他批次和其他知识库 |

## 附：RAG 核心编排全景

`DocumentProcessingService.processDocument()` 是整个数据准备阶段的编排入口：

```java
public void processDocument(String docId) {
    Document doc = documentRepository.findById(docId).orElseThrow(...);
    KnowledgeBase kb = knowledgeBaseRepository.findByKbIdAndIsDeletedFalse(doc.getKbId()).orElseThrow(...);

    // === 阶段1：文档解析（doc-parser 必需，无降级） ===
    updateStatus(docId, PARSING, 0.1f, "正在解析文档...");
    ParseResult parseResult = parseDocument(doc);

    // === 阶段2：文本分块（使用知识库配置的 chunkSize/overlap） ===
    updateStatus(docId, CHUNKING, 0.3f, "正在分块...");
    List<Chunk> chunks = createChunks(doc, parseResult, kb.getChunkSize(), kb.getChunkOverlap());
    chunkRepository.saveAll(chunks);

    // === 阶段3：向量化（使用知识库配置的 embeddingModel） ===
    updateStatus(docId, EMBEDDING, 0.6f, "正在向量化...");
    embedChunks(doc.getKbId(), chunks, kb.getEmbeddingModel());

    // === 完成 ===
    updateStatus(docId, COMPLETED, 1.0f, "处理完成");
}
```

这 3 个阶段 + 问答阶段的 `ChatService.sendMessage()`，构成了完整的 RAG 链路：

```
解析 → 分块 → 向量化 → 检索 → 生成
 ↑                        ↑
 │   数据准备（离线）       │    问答（在线）
 └────────────────────────┘
```

每个环节都是真实的，没有降级和 mock：
- 解析 → 必须依赖 doc-parser
- 分块 → 参数从数据库读取，前端可配置
- 向量化 → 调用真实的 Embedding API
- 检索 → 余弦相似度 + 阈值过滤，日志可见全过程
- 生成 → 防幻觉 Prompt + 对话历史 + 来源标注
