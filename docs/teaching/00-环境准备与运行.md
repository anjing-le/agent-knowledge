# 00 - 环境准备与运行

## 一、环境要求

| 组件 | 版本要求 | 用途 |
|------|---------|------|
| JDK | 17+ | 后端运行 |
| Maven | 3.8+ | 后端构建 |
| Node.js | 18+ | 前端运行 |
| pnpm | 8+ | 前端包管理 |
| MySQL | 8.0+ | 数据存储 |
| Python | 3.10+ | 文档解析服务（doc-parser，**必需**） |

## 二、API Key 配置

本项目需要两个 API Key，均使用 OpenAI 兼容接口（通过 OneRouter 代理）：

| 用途 | 默认模型 | 说明 |
|------|---------|------|
| Embedding | `text-embedding-3-small` | 将文本转为 1536 维向量 |
| LLM | `gpt-4o-mini` | 生成 RAG 回答 |

配置文件位置：`backend/src/main/resources/application-local.yml`

```yaml
app:
  embedding:
    api-url: https://llm.onerouter.pro/v1/embeddings
    api-key: sk-你的API密钥
    model: text-embedding-3-small
    dimensions: 1536

  llm:
    api-url: https://llm.onerouter.pro/v1/chat/completions
    api-key: sk-你的API密钥
    model: gpt-4o-mini
    max-tokens: 2048
    temperature: 0.7
```

> `application-local.yml` 已被 `.gitignore` 排除，不会上传到 GitHub。

## 三、数据库准备

```bash
# 登录 MySQL
mysql -u root -p

# 创建数据库
CREATE DATABASE agent_knowledge DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

数据库连接配置（`application-local.yml`）：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/agent_knowledge?useUnicode=true&characterEncoding=utf8&serverTimezone=GMT%2B8&useSSL=false&allowPublicKeyRetrieval=true
    username: root
    password: root    # ← 你的 MySQL 密码
```

> JPA 配置了 `ddl-auto: update`，首次启动会自动建表，无需手动执行 SQL。

## 四、doc-parser 文档解析服务（必需）

doc-parser 是独立的 Python 文档解析服务，支持 PDF、Word、PPT、Excel 等格式。

**本项目没有降级方案，doc-parser 必须启动**，否则文档上传后无法解析。

```bash
cd doc-parser

# 创建虚拟环境
python -m venv venv
source venv/bin/activate  # macOS/Linux

# 安装依赖
pip install -r requirements.txt

# 启动服务（端口 9001）
python -m uvicorn kparser.app:app --host 0.0.0.0 --port 9001
```

验证服务是否启动：
```bash
curl http://localhost:9001/health
# 返回 {"status": "ok"} 即成功
```

## 五、后端启动

```bash
cd backend

# 安装依赖
mvn clean install -DskipTests

# 启动应用（默认端口 10001）
mvn spring-boot:run
```

启动成功标志：

```
Started Application in x.xxx seconds
[启动] 开始重建内存向量索引...
[启动] 向量索引重建完成: 知识库=N个, 向量=N条
```

> 后端启动时会自动从数据库重建内存向量索引，之前上传的文档无需重新解析。

后端接口地址：`http://localhost:10001`

## 六、前端启动

```bash
cd frontend

# 安装依赖
pnpm install

# 启动开发服务器
pnpm dev
```

前端已配置 API 代理，`/api` 请求会自动转发到 `http://localhost:10001`。

## 七、验证方式

按以下步骤验证完整 RAG 链路：

### 步骤 1：创建知识库

打开前端页面 → 知识库列表 → 点击「新建知识库」 → 输入名称、描述 → 可配置 Embedding 模型、分块大小、重叠大小 → 确认

### 步骤 2：上传文档

进入知识库详情 → 查看顶部 RAG 配置栏（Embedding 模型、Chunk 大小、重叠） → 点击「上传文档」 → 等待状态变为 COMPLETED

### 步骤 3：查看切片

点击文档的「查看切片」 → 确认文档被正确分块

### 步骤 4：智能问答

进入「智能对话」页面 → 选择刚才的知识库 → 提一个文档中能找到答案的问题 → 查看 AI 回答是否标注了 ✅ 和来源文档 → 查看底部参考来源（包含文档名、相似度、引用原文）

### 步骤 5：验证 RAG 效果

- **对比实验**：新建一个不关联知识库的对话，问同样的问题 → AI 回答开头是 ⚠️，说明未使用知识库
- **去掉知识库**：在已关联知识库的对话中 x 掉知识库，再问问题 → 不走 RAG
- **不相关问题**：关联知识库后问一个文档里完全没有的问题 → AI 回答 ❌，不会编造内容

## 八、常见问题

### Q1：后端启动报数据库连接失败

检查 MySQL 是否启动，`application-local.yml` 中的用户名密码是否正确。

### Q2：上传文档后状态变为 PARSE_FAILED

doc-parser 服务未启动。本项目不提供降级方案，必须先启动 doc-parser。

### Q3：问答没有返回 RAG 结果

- 确认文档状态为 COMPLETED（向量化完成）
- 确认对话已关联知识库（左下角知识库标签不为空）
- 确认 Embedding API Key 配置正确
- 确认后端未重启（重启会自动重建向量，需要等启动日志显示"向量索引重建完成"）

### Q4：向量化失败（EMBEDDING_FAILED）

检查 Embedding API Key 是否有效，可以在终端测试：

```bash
curl https://llm.onerouter.pro/v1/embeddings \
  -H "Authorization: Bearer sk-你的密钥" \
  -H "Content-Type: application/json" \
  -d '{"model": "text-embedding-3-small", "input": "测试文本"}'
```

### Q5：重启后端后 RAG 检索不到结果

后端启动时会自动重建向量索引（从数据库读取已完成的 Chunk 重新 Embedding）。如果启动日志没有显示"向量索引重建完成"，说明知识库里没有已完成的文档，需要重新上传。

### Q6：前端页面空白或报错

- 确认 `pnpm install` 执行成功
- 确认后端服务已启动（前端依赖后端 API）
