深潜知识点

每个概念三层递进：比喻秒懂 → 技术准确 → 个人思考
1
RAG（检索增强生成）
▼
🎯 通俗比喻

RAG 就是让 AI 参加"开卷考试"。普通 ChatGPT 是闭卷——全靠记忆，记不住就编。RAG 是开卷——先翻书找到相关内容，再根据书上的内容回答问题，每个答案都有出处。

📖 技术本质

RAG 全称 Retrieval-Augmented Generation，分两步：第一步"检索"（Retrieval），把用户的问题和知识库里的文档做语义匹配，找出最相关的片段；第二步"生成"（Generation），把这些片段作为上下文塞进 Prompt，让大模型基于这些真实内容生成回答。核心价值是让大模型能回答它训练数据里没有的内容，同时减少幻觉。

💭 我的理解
我觉得 RAG 最精妙的地方在于它的"分工"——检索模块负责找对资料，生成模块负责组织语言。就像一个好的研究助手，先帮你把论文找出来摊在桌上，然后你再根据论文写综述。这样做的好处是知识更新很轻，我不需要重新训练模型，只要往知识库里上传新文档就行了。在做这个项目的过程中，我最大的体会是 RAG 看起来思路简单，但每个环节的质量都会影响最终效果——分块太粗检索不准，分块太细又丢失上下文，这些 trade-off 让我对 AI 工程有了更深的理解。
❓ 可能追问
Q: RAG 和直接把文档全塞给 GPT 有什么区别？
上下文窗口有限制（比如 128K Token），长文档塞不下；而且全塞进去检索效率很低、成本也高。RAG 先筛选出最相关的几个片段，精准且省钱。
Q: RAG 有什么局限性？
检索质量是瓶颈。如果用户问的方式和文档写法差异很大，向量检索可能召回不到正确的片段。可以通过 Rerank、Query 改写、混合检索来优化。
2
Embedding / 向量化
▼
🎯 通俗比喻

Embedding 就像是给每段文字发一个"GPS坐标"。不同的是这个坐标在一个1536维的空间里。意思相近的话，坐标就靠在一起——"退货条件"和"退款政策"的坐标很近，但和"今天天气"的坐标就隔得很远。

📖 技术本质

Embedding 是一种将离散的文本映射到连续的高维向量空间的技术。通过预训练的 Embedding 模型（如 text-embedding-3-small），将一段文字编码为1536维的浮点数向量。模型的训练目标是让语义相似的文本对应的向量在空间中距离更近（方向更一致）。

💭 我的理解
向量化让我最震撼的一点是——机器终于不是在做字符串匹配了，而是在理解"意思"。你搜"怎么退货"，它能找到写着"退款流程"的文档，哪怕两个句子没有一个字重复。在实现的时候，我注意到一个细节：问题向量化和文档向量化必须用同一个 Embedding 模型，否则生成的向量不在同一个空间里，算出来的相似度就没有意义了。这个项目里我把 Embedding 模型做成了知识库级别的配置项，保证同一个知识库下的文档和查询用的是同一个模型。
❓ 可能追问
Q: 1536维是什么概念？为什么需要这么多维？
维度越高，模型能编码的语义信息越丰富。就像用2个数字只能表示平面上的点，用3个能表示空间中的点。1536维是模型设计者在"信息容量"和"计算成本"之间选的平衡点。
Q: Embedding 模型是怎么训练出来的？
核心思路是对比学习。把语义相似的文本对（比如问答对）作为正样本，让模型学习把它们编码得更近；随机的文本对作为负样本，编码得更远。训练目标是让模型学会"什么是相似"。
3
余弦相似度
▼
🎯 通俗比喻

想象两支箭射出去——余弦相似度看的不是两支箭飞了多远，而是它们指向的方向是不是一致的。方向完全一致就是1，完全相反就是-1，毫无关系（垂直）就是0。在 RAG 里，我们用它来判断用户的问题和文档片段"聊的是不是同一件事"。

📖 技术本质

余弦相似度 = 两个向量的点积 ÷ 两个向量的模之积，即 cos(θ) = (A·B) / (|A|×|B|)。它衡量的是向量方向的一致性，范围 [-1, 1]。在文本检索中特别合适，因为不同长度的文本 Embedding 后向量的模可能差异很大，但余弦相似度只看方向，不受模长影响。

💭 我的理解
选余弦相似度而不是欧氏距离，是我在做的时候特意查过的。欧氏距离衡量的是"两点之间有多远"，余弦相似度衡量的是"两个方向有多像"。文本场景下，一篇长文章的向量模可能比一个短句子大很多，用欧氏距离会偏向短文本。但在语义上，长文章和短句子完全可以是"说的同一件事"，所以用余弦相似度更合理。
❓ 可能追问
Q: 余弦相似度 0.3 作为阈值是怎么定的？
经验值。太低（比如0.1）会引入不相关内容，太高（比如0.8）会错过措辞不同但有关的片段。0.3 在实测中过滤噪声效果不错。还在日志里打印了分数范围，方便调优。
4
文档分块策略
▼
🎯 通俗比喻

一本500页的书，你不会整本丢给别人让他找答案。你会先把书拆成一章一章、一段一段，然后告诉他"答案大概在第三章第二段"。分块就是把长文档拆成 AI 能处理的小段落的过程。

📖 技术本质

分块（Chunking）将长文本切分为固定大小的片段，每个片段独立向量化和检索。关键参数：chunk_size（每块多大，默认500字）和 overlap（相邻块的重叠区域，默认50字）。重叠防止语义在块边界处被截断——一句完整的话刚好在分界点，有了重叠，前后两个块都能包含完整语义。

💭 我的理解
分块大小是我反复调过的参数。太大的块检索定位不够精准——你想找"退货需要几天"，结果返回了一整页退货政策；太小的块丢失上下文——可能只有半句话。500字加50字重叠是试出来的平衡点。另外我在分块时还做了句子边界优化——尽量在句号或换行符处截断，而不是机械地按字数切。分块参数做成了知识库级别的配置，前端可以可视化编辑。
❓ 可能追问
Q: 除了固定窗口分块，还有什么策略？
语义分块（按段落/标题等自然边界切分）、递归分块（先粗切再细分）、针对结构化内容的专用分块策略。固定窗口最简单但最通用。
Q: 重叠太大或太小会怎样？
太大导致冗余高、向量库膨胀、检索时同一段内容反复出现。太小出现语义断裂。一般取块大小的10%左右作为重叠。
5
向量数据库 vs 传统数据库
▼
🎯 通俗比喻

传统数据库像图书馆的索引卡——按书名、作者精确查找，要么找到要么没有。向量数据库像一个"语义雷达"——你说"我想看关于太空探险的小说"，它不是按字面找，而是找到所有"意思接近"的书，按接近程度排序推荐。

📖 技术本质

传统关系型数据库（MySQL）存储结构化数据，用精确匹配做查询。向量数据库（Milvus、Qdrant）存储高维向量，用近似最近邻搜索（ANN）做查询——在海量向量中快速找到最相似的 Top-K。生产级向量数据库用 HNSW、IVF 等索引来加速。

💭 我的理解
在项目里我用了 ConcurrentHashMap 存向量，检索时暴力遍历。这是刻意的设计取舍——教学场景下数据量小（几百个片段），暴力搜索够用，而且能看到完整的相似度计算过程。但我清楚知道生产环境下不可行——百万级向量暴力搜索太慢，需要 ANN 索引。内存数据库重启就丢了，所以我实现了启动时自动重建的机制。
❓ 可能追问
Q: ANN 为什么叫"近似"？
精确最近邻要遍历所有向量（O(n)），太慢。ANN 用索引（如 HNSW 图结构）加速，速度快几百倍，但可能漏掉少量结果——是精度和速度的 trade-off。
6
幻觉问题（Hallucination）
▼
🎯 通俗比喻

幻觉就是 AI 在"一本正经地胡说八道"。你问它一个不知道的事情，它不会说"我不知道"，而是编一个听起来特别合理的答案。就像考试不会也不留白，硬写一个看起来像答案的东西。

📖 技术本质

幻觉产生的根源是语言模型的工作方式——本质上是在做"下一个词的概率预测"。它会生成统计上最可能出现的文本，而不是"最正确的"文本。当没有可靠信息时，它仍会生成流畅文本，因为流畅性是它被训练出来的核心能力。RAG 通过提供真实参考内容来约束模型。

💭 我的理解
幻觉是我最关注的问题。知识库问答如果出现幻觉，用户看到的是一个看起来很权威、标注了来源的错误答案，危害比普通聊天严重得多。我在 Prompt 里做了三层防护：第一"绝对禁止幻觉，忠于原文"；第二要求做相关性判断——不相关也要回答"❌ 未找到"；第三标注真实来源文档名让用户可以核实。效果比不加约束好很多，但不是百分百能杜绝，这是当前大模型架构的局限。
❓ 可能追问
Q: 除了 Prompt 约束，还有什么方法减少幻觉？
后处理校验（检查回答是否能在原文中找到）、Temperature 调低、多次生成取一致结果、让模型自我反思检查。还可以训练"拒绝回答"的能力。
7
Rerank（重排序）
▼
🎯 通俗比喻

向量检索像用渔网捞鱼——一网下去捞20条，有大鱼有小鱼还有水草。Rerank 就是一个经验丰富的渔夫，逐条看过来，把真正值钱的鱼挑出来排好序。第一步求快（能不能捞上来），第二步求准（是不是你要的）。

📖 技术本质

Rerank 使用交叉编码器（Cross-Encoder）对 query 和每个候选文档做精细打分。与 Embedding 检索的"双塔模型"不同——交叉编码器把 query 和文档拼在一起输入模型，能看到两者的交互信息，打分更准但速度更慢。典型流程：向量检索召回 Top-20 → Rerank 精排取 Top-5。

💭 我的理解
Rerank 我在项目里没有实现，但了解它的原理和价值。精妙之处在于"分层"思想——先用快但粗的方法圈定候选范围，再用慢但准的方法精选。这是信息检索的经典范式。我的 RetrievalService 流程是"向量搜索 → 阈值过滤 → 取TopK"，架构上已经预留了 Rerank 的位置。如果继续做下去，这是我最想加的功能。
❓ 可能追问
Q: 为什么交叉编码器比双塔模型更准？
双塔的 query 和文档独立编码，彼此"看不到对方"；交叉编码器把它们拼在一起，能捕捉细粒度交互（比如某个词在不同语境中的不同含义），所以更准确。代价是每对 query-doc 都要过一次模型。
8
Prompt Engineering（提示词工程）
▼
🎯 通俗比喻

Prompt 就是你"怎么跟 AI 说话"。同样的模型，随便问和精心设计问法，效果天差地别。就像同一个实习生，你说"帮我查下资料"和"帮我从这三份文件里找退货相关段落，找到后列要点，找不到就说没有"——第二种才能得到你想要的结果。

📖 技术本质

Prompt Engineering 通过设计输入文本来引导大模型产生期望输出。核心要素：角色设定（你是什么）、任务描述（做什么）、约束条件（不能做什么）、输出格式（怎么回答）、Few-shot 示例（给例子参照）。在 RAG 场景中还要包含检索到的参考内容和防幻觉指令。

💭 我的理解
Prompt 设计是投入时间最多的部分之一。刚开始写的 Prompt 很简单——"根据以下内容回答问题"，结果模型经常自由发挥，还编造不存在的来源名称。后来一版一版迭代，加入"绝对禁止幻觉"的最高优先级指令、用真实文档名做来源标注示例、不相关时的拒绝模板。最让我印象深刻的细节——如果在 Prompt 末尾再提醒一次"不要编造"，模型幻觉率会明显降低。所以在 System Prompt 结尾加了"再次提醒：只使用参考内容"。Prompt 设计是实验科学，要反复试。
❓ 可能追问
Q: System Prompt 和 User Prompt 的区别？
System Prompt 设定角色和行为准则，全程生效；User Prompt 是用户每次的具体问题。RAG 场景中 System Prompt 包含参考内容和防幻觉规则。
Q: Prompt 越长越好吗？
不是。太长消耗 Token，模型对超长 Prompt 的注意力会分散（"lost in the middle"）。要精简有力，把最重要的约束放在开头和结尾。

🔧 遇到的真实困难
困难1：@Async 事务问题
最头疼的一个 bug 是异步处理和数据库事务的冲突。我用 Spring 的 @Async 让文档处理在后台执行，结果异步线程里查不到刚插入的文档记录——因为事务还没提交呢，异步线程就开始跑了。查了很久才找到原因，最后用 TransactionSynchronizationManager 的 afterCommit 回调解决，确保事务提交后再触发异步处理。这个坑让我对 Spring 的事务机制理解深了很多。
困难2：向量重建策略
另一个挑战是内存向量库重启丢数据。一开始想"每次重启前把向量序列化到文件"，但向量是 Embedding API 生成的，序列化后如果 API 更新了模型版本，反序列化的向量可能和新生成的不一致。最后的方案是启动时从数据库重建——读出所有已处理的文档分块，批量重新调 Embedding API。虽然启动慢一点，但保证了向量的一致性。
困难3：Prompt 设计迭代
Prompt 也调了好多版。一开始模型经常编造不存在的文档名作为来源，后来我在 Prompt 里用检索到的真实文档名作为来源标注的示例，模型才不再乱编了。还有一个问题是检索到的内容和问题不相关时，模型还是会硬凑答案。加了"相关性判断"指令后效果非常明显。
🙋 "这代码是你写的吗？"
项目是我自己设计和实现的。开发过程中当然也参考了一些技术文档和开源项目的设计思路，比如 RAGFlow 和 Dify 的架构设计给了我很多启发。具体的代码实现、架构决策、参数调优都是我自己完成的，遇到的技术问题也是我自己排查解决的，比如刚才说的异步事务问题和 Prompt 迭代。
⭐ 坦诚但自信。承认参考了资料是正常的，重要的是你能说清楚为什么这样做、遇到了什么问题、怎么解决的。

9
Spring Boot 异步机制
▼
🎯 通俗比喻

你去奶茶店点单，同步模式是你站在柜台等奶茶做好再走；异步模式是店员给你取餐号，你先去逛街，做好了叫你回来取。文档处理可能要几分钟，不能让用户干等着。

📖 技术本质

Spring 的 @Async 注解让方法在独立线程中执行，调用方不阻塞。经典陷阱：在事务方法中调用 @Async 方法，异步线程可能在事务提交前执行，导致读不到当前事务写入的数据。解决方案是用 TransactionSynchronizationManager.registerSynchronization() 注册 afterCommit 回调。

💭 我的理解
这个坑踩得很深。文档上传接口里先创建 Document 记录保存到数据库，然后调 @Async 方法开始处理。结果异步线程里通过 documentId 查文档，查出来是 null——外层事务还没 commit。这个 bug 特别隐蔽，本地开发偶尔能查到（竞态条件），上了测试环境就频繁出问题。排查了两天才定位，最终用 afterCommit 回调解决。让我对 Spring 事务传播机制和线程隔离理解深了很多。
❓ 可能追问
Q: @Async 的线程池怎么配置的？
Spring 默认用 SimpleAsyncTaskExecutor（每次新建线程），生产环境应配 ThreadPoolTaskExecutor，指定核心线程数、最大线程数、队列容量。项目用默认配置，因为并发量不大。
Q: 异步任务失败了怎么办？
在 DocumentProcessingService 里做了 try-catch，失败时更新文档状态为 PARSE_FAILED / CHUNK_FAILED / EMBEDDING_FAILED，前端能看到失败状态。生产环境可加重试机制和死信队列。
10
JPA 实体设计
▼
🎯 通俗比喻

JPA 就是一个"翻译官"——你写 Java 类，它翻译成数据库表；你操作 Java 对象，它翻译成 SQL。不需要手写建表语句，也不用写大段 JDBC 代码。

📖 技术本质

JPA 是 Java 的 ORM 标准。核心是"实体映射"——用 @Entity、@Table、@Column 注解将 Java 类映射到数据库表。通过 ddl-auto=update 配置，JPA 可以自动更新表结构。Repository 接口继承 JpaRepository 后，CRUD 自动实现。

💭 我的理解
项目有五张核心表：KnowledgeBase、Document、Chunk、Conversation、Message，还有 FileStorage。JPA 省了大量写 SQL 的时间。实体之间的关系用外键字段而不是 @ManyToOne 注解维护，避免了 JPA 懒加载和 N+1 查询的坑——比如 Document 里存了 kbId 而不是 KnowledgeBase 对象引用。
❓ 可能追问
Q: JPA 的 N+1 问题是什么？怎么解决？
查列表时为每条记录的关联对象单独发一条 SQL，导致 1+N 条 SQL。方案：@EntityGraph 预加载、JPQL JOIN FETCH、或用外键字段避免对象关联。
Q: ddl-auto=update 在生产能用吗？
不建议。update 只加字段不删字段不改类型，可能导致不一致。生产用 Flyway 或 Liquibase 做数据库迁移管理。
11
RESTful API 设计
▼
🎯 通俗比喻

REST 就是用 HTTP 动词（GET/POST/PUT/DELETE）加名词性 URL 来描述对资源的操作。GET /api/books 就是"获取书的列表"，POST /api/books 就是"创建一本新书"，看到 URL 就知道在干什么。

💭 我的理解
API 设计遵循资源嵌套风格。知识库是顶层资源 /api/knowledge，文档是子资源 /api/knowledge/{kbId}/documents。对话模块单独 /api/chat，因为对话和知识库是多对多关系不适合嵌套。有一个设计做得不错——发消息时 kbIds 作为请求参数传入而非固定绑定在会话上，前端可以动态切换知识库，传空数组就变成纯 LLM 对话。
❓ 可能追问
Q: PUT 和 PATCH 有什么区别？
PUT 是全量更新（替换整个资源），PATCH 是部分更新（只改指定字段）。修改知识库名称用 PATCH 更合适。
12
前后端分离架构
▼
🎯 通俗比喻

就像餐厅的前厅和后厨。前厅（前端）负责接待客人、展示菜单；后厨（后端）负责做菜。两者通过"传菜口"（API）沟通，各管各的，前厅装修不影响后厨做菜。

💭 我的理解
前后端分离的好处很明显：前端用 Vite + Vue 3 热更新秒级生效，后端关注业务逻辑，互不影响。开发环境用 Vite 的 proxy 配置把 /api 前缀请求代理到 localhost:10001。Axios 封装做了统一的请求拦截（加 token）、响应拦截（处理错误码）、Loading 状态管理。
❓ 可能追问
Q: 跨域问题怎么解决的？
开发环境用 Vite devServer proxy。生产用 Nginx 反向代理，前端和 API 走同一域名的不同路径，就不存在跨域。
13
降级策略
▼
🎯 通俗比喻

降级就像电梯坏了改走楼梯——核心功能还能用，只是体验差一点。在 AI 系统里，如果 Embedding API 挂了，系统不能直接崩掉，应该有一个"差一点但能用"的备用方案。

💭 我的理解
我的项目刻意没做降级，这是设计决策。比如 doc-parser 不可用时直接返回错误。原因是教学项目希望每个环节都是真实的外部调用。但我清楚生产环境必须做降级——doc-parser 可降级为 Java 端 Apache Tika；向量检索可降级为 BM25 关键词搜索；LLM 挂了可直接返回原文片段。
❓ 可能追问
Q: 怎么判断什么时候该降级？
两种方式：主动健康检查（我的项目调用前先检查 /health），和被动熔断（连续 N 次失败后自动切到降级方案，比如 Resilience4j 的 CircuitBreaker）。
