第一轮：暖场开场 温和型

导师风格：微笑、鼓励、让学生放松 · 考察：表达能力、逻辑性
1
"简单介绍一下你自己和你的项目？"
▼
表达能力逻辑性
导师真实意图：这是破冰题，导师在观察你的表达是否清晰、是否紧张、能不能用简练的语言把一个技术项目讲明白。30 秒到 1 分钟最佳，讲太长会扣分。

✅ 参考回答

老师好，我叫 XXX，本科学的是软件工程。我做的项目是一个基于 RAG 技术的智能知识库系统。简单来说，就是让大模型能"读懂"企业自己的文档，然后基于文档内容来精准回答问题。用户上传 PDF、Word 这些文档后，系统会自动做文本提取、分块、向量化，存到向量库里。问答时先通过向量检索找到最相关的段落，再把这些段落交给大模型来生成回答，回答里还会标注内容来源。前端用的 Vue 3，后端 Spring Boot 3，文档解析是一个独立的 Python 服务。整个项目把 RAG 链路的每个环节都做了真实实现。

❌ 踩雷回答

  • "我做的是一个 AI 项目" — 太笼统，导师听完还是不知道你做了什么
  • 上来就背技术栈清单："Spring Boot、Vue、MySQL、FastAPI、OpenAI..." — 听起来像在念简历，没有逻辑
  • 讲了 3 分钟还没讲完 — 暖场题需要简洁，讲太久导师会不耐烦

🛟 救场技巧

如果紧张忘词，抓住一个核心公式："我做的是 XX 系统，它解决了 XX 问题，用的是 XX 技术方案"。三句话就够了，后面导师自然会追问细节。

学术潜力表达能力

✅ 参考回答

最初是用 ChatGPT 的时候发现一个问题——它对通用知识回答得很好，但问我自己公司的产品手册、内部规范这些东西时就完全不行，要么说不知道，要么编一个看起来对但其实不对的答案。我当时就好奇：能不能让大模型先"读"我的文档，再基于文档来回答？后来了解到 RAG 就是解决这个问题的，觉得特别有意思。它不像微调那样需要重新训练模型，而是通过检索的方式把外部知识注入进去，这个思路很巧妙。做完项目之后我对文档解析、向量化、检索这些环节都有了实际体验，越做越觉得里面可以优化的地方很多。

❌ 踩雷回答

  • "因为 AI 现在很火" — 导师最讨厌的回答之一，说明你没有自己的思考
  • "因为好找工作/好发论文" — 太功利，导师会觉得你读研动机不纯
  • "我也不太确定，就是想试试" — 显得没有目标感

🛟 救场技巧

关键是讲一个具体的触发点——是什么事情、什么场景让你对这个方向产生了好奇。哪怕是"我用 ChatGPT 的时候发现它不能回答我的文档内容"这种很小的事也行，比空洞的"因为感兴趣"强一百倍。

"做这个项目花了多久？过程顺利吗？"

诚实度表达能力
导师真实意图：看你的项目经历是否真实。太顺利会显得假，完全不顺利又会显得能力不够。导师希望听到"遇到了什么问题，怎么解决的"。

✅ 参考回答

前前后后大概用了两个多月。过程不算特别顺利，踩了不少坑。印象最深的是异步处理那块——文档上传后需要异步做解析、分块、向量化，我一开始用 Spring 的 @Async，结果发现异步线程跑起来的时候事务还没提交，导致查不到刚插入的文档记录。后来用了事务同步回调的方式，确保事务提交后再触发异步任务才解决。还有向量检索的阈值调优也花了不少时间，最开始设得太高很多相关内容被过滤掉了，后来反复试了不同的值，最终定在 0.3。

❌ 踩雷回答

  • "花了一周就做完了" — 导师会怀疑你只是跑了个 demo

🛟 救场技巧

准备 2-3 个"我踩过的坑"故事，每个故事包含问题→排查过程→解决方案。这比任何技术名词都更能证明你真的写过代码。

🔮 导师可能的追问

第二轮：技术深入 探究型

导师风格：好奇、追问细节 · 考察：技术理解深度、是否真正动手做过
1
"RAG 的完整流程能给我讲一下吗？"
▼
技术深度逻辑性
导师真实意图：这是核心考题。导师想看你能不能把整个链路串起来讲清楚，而不是只知道某个环节。能清晰分"离线"和"在线"两条线讲的学生，会给导师留下很好的印象。

✅ 参考回答

RAG 分两个阶段。第一个是数据准备阶段，也叫离线阶段，就是把文档预处理好：先通过文档解析把 PDF、Word 这些格式的文件提取成纯文本；然后做文本分块，把长文本按固定窗口切成比如 500 字一段的小块，相邻块重叠 50 字防止语义截断；接着用 Embedding 模型把每个小块转成 1536 维的向量；最后存进向量库。这套流程每个文档只需要跑一次。

第二个是问答阶段，也叫在线阶段：用户提问后，先把问题也用同一个 Embedding 模型转成向量，然后在向量库里用余弦相似度找到和问题最相关的 Top-5 段落，过滤掉相似度低于 0.3 的噪音，然后把这些段落作为参考内容拼进 System Prompt，连同对话历史一起发给大模型，让它基于这些参考内容生成回答。回答会标注来源文档名，方便用户核实。

❌ 踩雷回答

  • "就是先检索再生成" — 太笼统，导师会觉得你根本没做过
  • 只讲在线阶段不讲离线 — 说明你不理解数据准备的重要性
  • 讲的时候逻辑混乱，来回跳 — 按"离线→在线"的顺序讲，不要跳着说

🛟 救场技巧

记住这个口诀："解析→分块→向量化→入库，提问→向量化→检索→拼 Prompt→生成"。先背住流程骨架，细节导师会追问的。

🔮 导师可能的追问

  • "离线和在线的 Embedding 模型必须一样吗？为什么？"
  • "向量化的维度是多少？为什么是这个数？"
2
"向量化用的什么模型？为什么选这个？维度是多少？"
▼
技术深度
导师真实意图：看你对自己用的技术是不是真了解。很多学生能说出模型名字但不知道维度、不知道为什么选它。

✅ 参考回答

用的是 OpenAI 的 text-embedding-3-small，输出 1536 维的向量。选它主要是三个考虑：第一，性价比高，small 版本价格是 large 的 1/5 但效果对 RAG 场景已经够用了；第二，接口标准是 OpenAI 兼容的，后面如果要换成其他厂商的模型，只要接口兼容就能直接替换，我项目里的 EmbeddingService 已经做了模型可配置；第三，1536 维在信息容量和计算成本之间是一个比较均衡的选择。

❌ 踩雷回答

  • "用的 OpenAI 的模型" — 说不出具体名字，说明你没有真正配置过
  • "维度我不太记得了" — 1536 是核心参数，这个必须记住

🛟 救场技巧

如果忘了具体维度，可以说"具体维度我需要看一下配置文件，但我知道维度越高信息越丰富，同时计算成本也越大，我选的是性价比较高的 small 版本"。展示你理解维度的含义比记住数字更重要。

🔮 导师可能的追问

  • "Embedding 模型的训练原理是什么？它怎么学到语义的？"
  • "你知道 BGE 或者其他中文优化的 Embedding 模型吗？"
3
"余弦相似度的公式你能写出来吗？为什么用余弦不用欧氏距离？"
▼
技术深度
导师真实意图：这是区分"背答案"和"真理解"的关键题。能写出公式说明你学了，能解释"为什么不用欧氏距离"说明你真的想过。

✅ 参考回答

公式是 cos(θ) = A·B / (|A| × |B|)，展开就是所有维度的 aᵢ×bᵢ 求和，除以两个向量各自的模。用余弦而不用欧氏距离，是因为余弦相似度只看两个向量的方向是否一致，不受向量长度的影响。这对文本场景很重要——同一个主题的短文本和长文本，语义是接近的，但由于长度不同，Embedding 向量的模可能差很多。如果用欧氏距离，长短文本的端点距离会很大，即使它们方向一致也会被判断为不相似；但余弦相似度只看角度，不管长短，所以更适合文本检索。我项目里 VectorStoreService 的 cosineSimilarity 方法就是按这个公式实现的。

❌ 踩雷回答

  • "余弦相似度就是算两个向量的相似度" — 等于没说，导师要的是公式和原理
  • "欧氏距离也可以用吧" — 说明你不理解两者的本质区别

🛟 救场技巧

即使公式没记全，抓住核心："余弦看方向，欧氏看距离。文本场景下方向比距离更能反映语义相似性，因为文本长短不同但说的是同一件事"。这个直觉比背公式更有价值。

🔮 导师可能的追问

  • "那内积呢？内积和余弦的关系是什么？"
  • "如果向量都做了归一化，余弦和欧氏距离有什么关系？"
4
"分块策略怎么设计的？重叠是干嘛的？"
▼
技术深度逻辑性
导师真实意图：分块是 RAG 里容易被忽视但很关键的环节。导师想看你是不是真的思考过"为什么要这么切"。

✅ 参考回答

我用的是固定窗口加重叠的策略，默认每块 500 字，重叠 50 字。重叠的作用是防止语义截断——假如一句完整的话刚好在分界点被切成两半，前一块缺后半句，后一块缺前半句，检索时两块都匹配不好。加了重叠后，这句话会同时出现在前后两块里，至少有一块能完整匹配到。另外代码里还做了句子边界优化，尽量在句号或换行处截断而不是在字的中间硬切。这些参数都是可配置的，存在知识库实体里，前端也能编辑。

❌ 踩雷回答

  • "就是 500 字一切" — 完全没提重叠和边界优化，说明只是用了默认值没思考过
  • 说不出重叠的作用 — 这是面试高频考点，必须会

🛟 救场技巧

用一个比喻："就像切香肠，每段都留一点在下一段的开头，这样任何一刀都不会把一句完整的话切断。"

🔮 导师可能的追问

  • "块太大和块太小分别有什么问题？"
  • "500 字这个值怎么定的？有没有做过消融实验？"
5
"你的检索阈值设的多少？怎么定的这个值？"
▼
技术深度诚实度
导师真实意图：这个问题能暴露你是不是真的调试过检索环节。只会说"0.3"不知道怎么来的，说明你没有调过参数。

✅ 参考回答

默认设的 0.3。这个值是通过观察实际检索结果的分数分布来定的。我在 RetrievalService 里加了日志，每次检索会打印过滤前的分数范围和数量，比如"候选 20 条，分数区间 0.21-0.48，阈值 0.3，过滤后 5 条"。一开始我试过 0.5，发现很多实际相关的内容分数只有 0.35-0.45 也被过滤掉了，召回率太低。后来降到 0.3，在"不漏掉相关内容"和"不引入太多噪音"之间取了一个平衡。这个值其实跟 Embedding 模型和文档类型都有关系，不同场景可能需要重新调。

❌ 踩雷回答

  • "0.3 是网上推荐的" — 诚实但说明你没有自己验证过
  • "阈值？什么阈值？" — 说明你不了解自己项目的检索环节

🛟 救场技巧

"这个值我参考了业界常用的范围，然后在自己的测试文档上做了几轮验证。我在日志里打印了每次检索的分数分布，根据实际效果做了调整。"——即使你确实参考了网上的值，加上"我验证了"会好很多。

🔮 导师可能的追问

  • "如果阈值过高会怎样？过低呢？"
  • "你有没有想过用自动化的方式来确定阈值？"
📍

导师心理转场：学生技术基础不错，能讲清流程和原理。但技术选型是不是深思熟虑的？还是说只是因为教程这么写就这么做了？开始挑战决策背后的 trade-off。

第三轮：架构决策追问 挑战型

导师风格：质疑、追问 why not · 考察：思考深度、trade-off 意识
1
"为什么用内存存向量而不用 Milvus？你知道 Milvus 吗？"
▼
技术深度思维广度
导师真实意图：看你是否知道"正确答案"但有意做了简化，还是根本不知道有更好的方案。能解释 trade-off 的学生比只会用最优方案的学生更有深度。

✅ 参考回答

知道 Milvus，它是一个专业的向量数据库，支持十亿级别的向量检索，用 HNSW 或 IVF 等近似最近邻算法，检索速度是 O(log n) 级别的。我项目里没用它，是基于两个考虑：第一，项目面向的是教学场景，数据量通常在几千到几万条，ConcurrentHashMap 暴力遍历的性能完全够用，而且学生能直接在代码里看到余弦相似度是怎么一行行算出来的；第二，用内存存储意味着重启会丢数据，我专门写了一个 VectorIndexRebuilder，在应用启动时自动从数据库查出所有已向量化的分块，重新调 Embedding API 生成向量写入内存。这个"发现问题→设计解决方案"的过程本身也是值得讲的。当然，如果数据量到了百万级别，那一定要换成 Milvus 或 Qdrant。

❌ 踩雷回答

  • "Milvus 是什么？" — 完全不知道主流向量数据库，知识面太窄
  • "没有用 Milvus 因为太复杂了" — 虽然诚实但说法太消极，改成"出于 XX 考虑做了简化"

🛟 救场技巧

核心话术："我知道生产环境应该用 XX，但在当前场景下我选择了 YY，因为……"。先展示你知道"正确答案"，再解释为什么当前做了不同选择，最后说"如果要扩展会怎么做"。

🔮 导师可能的追问

  • "VectorIndexRebuilder 具体怎么实现的？用的什么触发机制？"
  • "暴力遍历的时间复杂度是多少？和 HNSW 比呢？"
2
"文档解析为什么不在 Java 里做，非要用 Python 单独起服务？"
▼
技术深度逻辑性
导师真实意图：微服务拆分的合理性。导师想看你是不是"为了用 Python 而用"，还是有合理的技术判断。

✅ 参考回答

主要是 Python 在文档解析这块的生态确实比 Java 好很多。PDF 解析有 PyMuPDF、pdfplumber，Word 解析有 python-docx，如果后面要做 OCR 还有 Tesseract 的 Python binding。Java 虽然有 Apache Tika，但灵活度和社区活跃度不如 Python。另外独立服务还有一个好处——文档解析的依赖比较重（各种 native 库），放在 Java 应用里会把主服务搞得很臃肿。拆成独立服务后，两边可以独立部署、独立升级，解析服务挂了也不影响主业务。当然代价是多了一次网络调用和服务依赖管理，我在调用前加了健康检查。

❌ 踩雷回答

  • "因为我 Python 比较熟" — 这是个人原因不是技术原因
  • "微服务架构嘛，就应该拆" — 拆服务需要理由，不是"应该拆"

🛟 救场技巧

从生态优势和关注点分离两个角度回答。前者是技术理由（Python 的解析库更强），后者是工程理由（解析和业务逻辑解耦）。

🔮 导师可能的追问

  • "两个服务之间怎么通信的？用的什么协议？"
  • "如果 doc-parser 挂了怎么办？有降级方案吗？"
3
"LLM 调用为什么是同步的不做流式？用户体验不差吗？"
▼
技术深度诚实度
导师真实意图：看你是否了解流式输出（SSE），以及是否能诚实承认不足并给出改进思路。

✅ 参考回答

确实，当前的实现是同步的，用户需要等几秒才能看到完整回答，体验不如 ChatGPT 那种逐字输出的效果。没做流式主要是时间和精力的权衡——流式输出需要后端用 SSE（Server-Sent Events）推流，前端做流式渲染和打字机效果，链路比较长。但我知道实现思路：后端用 WebFlux 或 SseEmitter 接收 LLM 的流式 response，每收到一个 token 就推给前端，前端用 EventSource 接收并逐字显示。如果继续做这个项目，流式输出是我优先级最高的优化项之一。

✅ 参考回答

最大的收获是理解了"大模型应用和传统应用的本质区别"。传统应用是确定性的——相同输入一定相同输出，可以写精确的单元测试。但大模型应用的输出是概率性的，每次可能不一样，而且可能出错。这意味着你不能只靠模型本身，必须在工程层面做大量的保障——Prompt 约束、检索兜底、来源标注、阈值过滤。做完这个项目我才真正体会到，大模型应用开发的难点不在于调 API，而在于怎么让不确定的系统变得可靠。这个认知对我后续的学习和研究方向影响很大。

🛡️ 面试生存指南

1

不会就说"这块我了解不深，但我的理解是…"

诚实 > 装懂。导师更尊重知道自己边界在哪里的学生。说"不知道"之后紧接着给出你的推测或思考方向，比瞎编一个答案强十倍。

2

每个技术选型都要能说出"为什么选它"和"它的不足是什么"

导师最欣赏有 trade-off 意识的学生。不是选了什么重要，而是为什么选它、放弃了什么、如果要改进会怎么做。这才是研究生思维。

3

准备 3 个"我踩过的坑"故事

比如 @Async 事务问题、阈值调优过程、重启数据丢失的解决方案。每个故事包含"问题→排查→解决"三步。这是最有说服力的"我真的做过"的证据。

4

面对压力题，先深呼吸，再用"承认+但是"话术

"确实，这块是项目的不足，但我了解改进方向是 XX，如果继续做我会 XX"。不卑不亢，既承认不足又展示思考。

5

回答要有"信号量"——让导师知道你的层次

提到一个你"知道但没做"的技术时，用这个句式："我实现的是方案 A，但我了解方案 B 的原理，它的优势是…如果要优化我会考虑引入。"这句话同时展示了动手能力、知识面和思考深度。
